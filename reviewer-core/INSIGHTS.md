## Codebase Patterns
<!-- Conventions & architectural decisions, with the "why". -->
- **CI fail-gate parity across studio ingest, runner exit code, and GitHub review events requires one counts-based API in this package — never a parallel rank map in the server.** SPEC-05 initially reimplemented severity×`ci_fail_on` in `server/src/modules/ci/verdict.ts` with a local `COUNT_SEV_RANK` while the runner used `gateTriggered(findings, …)` and `artifact.ts` counted severities separately — architecture-reviewer High: the three can drift. Fix: export `severityCounts` + `gateTriggeredFromCounts` from `src/output/to-review.ts:41,62` next to `FAIL_ON_MIN_RANK` / `SEV_RANK`, and make `gateTriggered` (`:75`) delegate to the counts path so finding-based and count-based callers cannot disagree. Studio `ciRunStatus` and agent-runner artifact building import these; do not reintroduce a third rank table in server or agent-runner.
- **Adding a new prompt section (omit-when-empty, byte-identical baseline).** Three steps: (1) add the optional field to `ReviewInput` in `src/review/run.ts` and thread it through the single `promptParts` object (`src/review/run.ts:130`) — that object feeds BOTH the whole-diff assembly AND the per-chunk loop, so threading it once covers both paths; (2) in `src/prompt.ts` build the section only when the field is present and push it into `userSections` only when truthy (so an absent field changes nothing); (3) add a matching `.nullish()` field to `PromptAssembly` in the vendored `trace.ts` — in BOTH `server/` and `client/` copies (no auto-sync) — or the assembly object won't typecheck. `test/prompt.test.ts` guards the no-section path as byte-identical to the old prompt — run it after. Worked example: the `intent` slot (`## Declared intent` = `wrapUntrusted('intent', …)` followed by a TRUSTED, unwrapped scope rule) in `src/prompt.ts`.

## Tool & Library Notes
<!-- Dependency quirks, version gotchas, env/config oddities. -->
- **`pnpm run build` / `pnpm run typecheck` / `pnpm exec tsc` in this package can all fail with `[ERR_PNPM_IGNORED_BUILDS] Ignored build scripts: esbuild@0.21.5, esbuild@0.28.1` before `tsc` even runs — a supply-chain gate on `esbuild`'s postinstall script, unrelated to any code change.** `pnpm exec`/`pnpm run` re-triggers pnpm's dependency-status check on every invocation, and it refuses to proceed while those builds are pending approval (`pnpm approve-builds` fixes it permanently but prompts interactively / mutates project config — not appropriate mid-task). Fastest workaround that avoids both: call the installed binary directly, bypassing pnpm's wrapper — `./node_modules/.bin/tsc --noEmit -p tsconfig.json` — which typechecks cleanly with zero output on success and skips the install-gate entirely.
- **OpenAI's `zodResponseFormat` → `toJsonSchema` (`src/llm/structured.ts`) emits `$ref` + `#/definitions/…` whenever the same Zod enum/object is reused in multiple places.** For `Review` (`findings.ts`), the shared `TrifectaComponent` enum collapses into one definition referenced from both `findings[].trifecta_components[]` and `findings[].evidence[].component`. OpenAI-compatible providers (DeepSeek via OpenRouter) accept that; **Google Gemini (e.g. `google/gemini-2.5-flash-lite`) rejects it** as `Invalid response_json_schema: reference to undefined schema at …evidence…component`. Plain chat and a hand-written schema *without* `$ref` both succeed on the same model — the failure mode is specifically `$ref` under `response_format: json_schema` + `strict: true` in `OpenRouterProvider.completeStructured` (`src/llm/openrouter.ts`). When adding shared Zod enums used in structured LLM output, prefer inlining (or dereference `$ref`s before send) if Gemini must stay in the model picker.
- **(2026-07-15 correction of the `$ref` entry above)** Dedup is not “both fields `$ref`” — dump of `toJsonSchema(Review)` shows the **first** use of `TrifectaComponent` (`trifecta_components.items`) stays **fully inlined** (`type`+`enum`), while only the **second** use (`evidence[].component`) becomes `{ "$ref": "#/definitions/…" }` with the shared body under top-level `definitions`. Gemini’s “undefined schema” error therefore points at `…evidence…component` even though a schema reader glancing at `trifecta_components` sees no `$ref`. Not “Gemini can’t do enums” or “can’t do structured JSON” — same model accepts an equivalent schema with the enum copied inlined at both sites. Google’s structured-output path simply does not resolve JSON Schema `$ref`/`definitions` (widely reported across Gemini API / tool schemas); DeepSeek’s OpenAI-compatible path does.

## Recurring Errors & Fixes
<!-- Error signature → root cause → fix. -->
- **`400 Provider returned error` on every eval/review after switching an agent to a Gemini model via OpenRouter** → OpenRouter metadata `raw` says Google `INVALID_ARGUMENT`: `reference to undefined schema at properties.findings.items.properties.evidence.anyOf.0.items.properties.component`. Root cause: `toJsonSchema(Review)` `$ref`s (see Tool & Library Notes) that Gemini does not resolve. Symptom timing is a tell: HTTP 200 from `POST …/eval-runs` in <1s with `actual_output` a string error and null metrics — not a slow scoring miss. Immediate unblock: switch the agent back to a non-Gemini OpenRouter model (e.g. `deepseek/deepseek-v4-flash`). Proper fix: dereference/`$ref`-inline the JSON schema (or stop sharing Zod enums across sibling fields) before `chat.completions.create`. Confirmed live 2026-07-15 on agent `19631087-…` v22 (`google/gemini-2.5-flash-lite`); same `completeStructured` path succeeds on DeepSeek with the identical schema.
