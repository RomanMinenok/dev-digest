## What Works
<!-- Approaches/solutions that worked. e.g. "X via Y in src/foo.ts:42 because …" -->
- Computing PR-list cost with a separate `SELECT cost_usd FROM agent_runs WHERE status='done' ORDER BY ran_at DESC` + JS grouping (same pattern as score in `modules/pulls/routes.ts:119-130`) — cheap enough for a small PR list, keeps the query readable, and avoids a complex JOIN.
- Spreading `{ usage: { include: true } }` into the OpenAI SDK `chat.completions.create()` call DOES reach OpenRouter — the SDK passes untyped extra body fields through without stripping them. Confirmed: `res.usage.cost` is populated for both plain completions and structured output (`response_format: json_schema`) calls. The cast `(res.usage as { cost?: number })?.cost` is correct.

## What Doesn't Work
<!-- Dead ends & antipatterns. The most valuable & most-skipped section. -->
- **`polling/routes.ts` `onConflictDoUpdate` only updates what you explicitly list — diff-size fields were missing and silently stayed stale.** The original set included `title`, `headSha`, `status`, `updatedAt` but omitted `additions`, `deletions`, `filesCount`. Those three are written on initial INSERT (new PRs) but never refreshed, so the SIZE column showed permanently stale values even after a poll. Fix: add them explicitly to the conflict set (`polling/routes.ts:50-60`). Rule: whenever adding a new column to `pull_requests`, check whether it belongs in the conflict-update set too.
- **New `FEATURE_MODELS` entries must default to `openrouter`, not `openai`** — the `conventions` entry shipped with `defaultProvider: 'openai'`/`defaultModel: 'gpt-5.4'`, which silently demands `OPENAI_API_KEY` at runtime even when the workspace has only `OPENROUTER_API_KEY` configured. Every other feature (`onboarding`, etc.) defaults to `openrouter`. Check `FEATURE_MODELS` in both vendor copies of `platform.ts` (`server/src/vendor/shared/contracts/platform.ts` and `client/src/vendor/shared/contracts/platform.ts`) when adding a new feature — the default propagates directly to `resolveFeatureModel` → `container.llm(provider)`.
- "Take the single newest completed `agent_run` for PR-list cost" is wrong when multiple agents run together — you get only one agent's cost. The correct definition is **sum of the latest run per agent** (newest `ran_at` per `agent_id`, then sum `cost_usd`). Implemented in `modules/pulls/routes.ts` via a `seenAgentPerPr` Set + accumulator loop over rows sorted `ran_at DESC`.
- `run-executor.ts` was calling `priceBook.estimate()` unconditionally, discarding `outcome.costUsd` that the engine already computed from the OpenRouter API. The engine (`reviewPullRequest` in `reviewer-core`) returns `costUsd` on the `ReviewOutcome` — always prefer it; fall back to estimate only when null: `outcome.costUsd ?? priceBook.estimate(agent.model, tokensIn, tokensOut)` (fixed in `modules/reviews/run-executor.ts:214`).
- **(2026-06-19 correction of above)** "Sum of latest run per agent" is also wrong — it mixes agents from different review sessions (e.g. Security from yesterday + General from today). The correct approach is a **60-second session window**: find the newest `ran_at` for the PR, then sum `cost_usd` of all completed runs within 60 s of it. Agents launched together by one "Run Review" click start within milliseconds; 60 s safely covers any DB round-trip lag. Implemented in `modules/pulls/routes.ts` (two-pass: find `latestRanAtByPr`, then filter + sum).

## Codebase Patterns
<!-- Conventions & architectural decisions, with the "why". -->
- **`SEED_SKILLS` (`db/seed-skills.ts`) and `agentSkillLinks` (`db/seed.ts`) are two independent lists with no cross-check** — nothing enforces that every seeded agent gets a linked skill, or that the skill count meets a course-lesson minimum. Found via review feedback: only 4 skills existed (course required ≥6) and 3 of the 5 seed agents (`General Reviewer`, `Security Reviewer`, `Performance Reviewer`) had zero entries in `agentSkillLinks` despite their `description` implying skill-driven behavior. Fixed by adding `security-rubric`/`performance-rubric` to `SEED_SKILLS` and a link per previously-unlinked agent. When adding a new seed agent or skill, manually verify both lists — there is no test asserting "every agent has ≥1 linked skill" or a minimum skill count.
- `ReviewRepository` in `modules/reviews/repository.ts` is a class wrapper around the function-level repos in `repository/run.repo.ts`, `repository/review.repo.ts`, etc. When adding a field to `completeAgentRun`, you must update **both** the underlying function signature in `run.repo.ts` AND the class method signature in `repository.ts` — a mismatch causes a TS error at the `run-executor.ts` call site, not at the definition, which is confusing.
- **Linked skills reach a review run ONLY through `run-executor.ts`, not the engine or the agents module.** The engine (`reviewPullRequest`) and `assemblePrompt` already accept a resolved `skills: string[]`, but until `run-executor.ts` loads them, linking a skill to an agent has zero effect on a review. The executor loads them via its constructor-injected agents repo (`this.agents.linkedSkills(agent.id)` = `container.agentsRepo`, already ordered by `agent_skills.order`), filters to `skill.enabled`, and passes `...(skillBodies.length ? { skills } : {})` (omit-when-empty, like `callers`/`repoMap`). The in-prompt gate is therefore **linked AND `enabled`**, enforced at that filter — not in the engine. Skill bodies are passed as trusted instructions (NOT `wrapUntrusted`); the trust mitigation is import-disabled-by-default + manual vetting.
- `agent_runs` has **no direct FK to `findings`**. The join chain for per-run severity counts is `findings.review_id → reviews.id → reviews.run_id`. To query severity breakdown per run: `JOIN reviews ON findings.review_id = reviews.id WHERE reviews.run_id IN (...)` grouped by `reviews.run_id, findings.severity`. There is no shortcut via `agent_runs` — the link only exists through `reviews`. See `modules/reviews/repository/run.repo.ts:listRunsForPull` and `modules/pulls/routes.ts` for the implemented pattern.
- **`@devdigest/shared` is vendored in two places with no automated sync.** `server/src/vendor/shared/` is the canonical copy; `client/src/vendor/shared/` is a manual mirror. TypeScript will NOT catch drift between them — each package resolves `@devdigest/shared` to its own local copy, so both sides compile independently even when the schemas diverge. Any edit to a contract file must be applied to both directories. The breach patterns and detection commands for drift are in `docs/skills/contract_breach.md`.
- **Anti-hallucination gate for LLM extraction**: after `completeStructured`, verify each candidate by taking `evidence_snippet.split('\n')[0].trim()` and checking the referenced file on disk literally `includes()` that string. Drop the candidate if the file is missing or the first line is absent. This kills hallucinated `file:line` references without fuzzy matching or embeddings. Implemented in `modules/conventions/extractor.ts`; re-use this pattern for any future LLM feature that returns evidence paths.
- **`repoIntel.getConventionSamples()` returns `[]` when the index is disabled or not yet built** (the `RepoIntel` degraded contract). LLM features that need sample source files must have a fallback or the scan silently operates on zero files. Convention extractor handles this with `fallbackWalk` in `modules/conventions/helpers.ts` — a simple BFS over the clone filtered to `.ts/.tsx/.js/.jsx`, excluding `node_modules`/`dist`/`.git`/migrations/test files. Apply the same pattern for future features that consume `getConventionSamples` or similar `RepoIntel` methods.
- **The persisted `references` table (repo-intel's caller graph, `getResolvedCallers` in `modules/repo-intel/repository.ts:503`) does NOT resolve calls made through an interface-typed property back to the concrete implementing class's method.** `LocalNoAuthProvider.currentUser`/`currentWorkspace` (`adapters/auth/local.ts`) are called ~30 times as `container.auth.currentUser(req)` (`container.auth` is typed `AuthProvider`), yet a direct query — `SELECT count(*) FROM "references" WHERE decl_file = 'server/src/adapters/auth/local.ts' AND to_symbol = 'currentUser'` — returns **0**. The indexer's reference resolution matches call sites to declarations by name+file heuristics that don't cross an interface/DI boundary, so any method only ever invoked via `container.<interface>.method(...)` will show as having 0 callers in Blast Radius even though it's very much alive. Don't read "0 callers" in Blast Radius as "dead code" for anything reached through an injected interface (`container.auth`, `container.repoIntel`, `container.llm`, etc.) — check real call sites with a plain grep before concluding a symbol is unused.
- **`GitClient.readFile(repo, path)` reads the clone's CURRENT working-tree checkout — NOT a specific sha; there is no head-pinned read API** (`@devdigest/shared` `adapters.ts` `GitClient`). To read a repo file "at the PR head" you must sync the clone to that head first — only `modules/repo-intel/service.ts:151` does an explicit `git.sync`; the review diff path (`loadDiff`) just uses whatever is checked out and falls back to persisted `pr_files` patches. So `modules/intent/spec-resolver.ts` (wired at `modules/intent/service.ts:84`) reads repo `.md` from the current checkout, which is not guaranteed to be the PR head. If you write a feature that must read repo files at a precise commit, sync first — don't trust a `readFile(sha, …)` signature (it doesn't exist; plan prose that says so is wrong).
- **Unit-testing a service method that mixes DB reads with pure composition logic: extract the pure part into an exported function taking plain data, rather than mocking Drizzle's chainable query builder.** `PullsService.smartDiff` needed tests for session-window filtering, classification grouping, and split-threshold logic (Decision 6 in the Smart Diff plan) but also does two `container.db.select().from().where()` calls and instantiates `ReviewService`. Mocking that chain (or `ReviewService`) for a plain unit test is awkward and brittle. Fix: extracted `assembleSmartDiff(prFiles, reviews)` as a pure function in `modules/pulls/service.ts` — `smartDiff()` just fetches and delegates to it. Tests call `assembleSmartDiff` directly with in-memory fixtures, no mocking needed. Reuse this split (thin DB-fetching method + pure composition function) for any future service method that needs unit tests without Testcontainers.
- **`pulls/service.ts` is the only `service.ts` in the codebase that talks to Drizzle directly (`import { and, eq } from 'drizzle-orm'` + `container.db.select().from(t.pullRequests)...`) instead of going through a `repository.ts`.** Every other module with a service (`reviews`, `agents`, `conventions`, `intent`, `repo-intel`, `repos`, `skills`) routes DB access through a repository. This is a deliberate, scoped exception (Smart Diff plan Decision 1 — "repository.ts is not added ... matching the module's current pre-existing, non-canonical style rather than introducing a disproportionate refactor"), confirmed by an architecture review as sound-with-risk (Medium severity) rather than a defect. Don't copy `pulls/service.ts`'s direct-Drizzle style as a template for a new module's `service.ts` — extract a `repository.ts` instead, matching the other 7.
- **`PinoLike` (`platform/run-logger.ts:21`) is the canonical minimal structural logger type** (`(obj, msg?)`, pino-compatible). `modules/reviews/run-executor.ts:21` has an identical duplicate exported as `Logger`. A new module that needs a logger param should import `PinoLike` from `platform/run-logger`, NOT `Logger` from `modules/reviews/run-executor` — the latter creates a wrong-direction `module → reviews` import edge (flagged in the intent architecture review; fixed at `modules/intent/service.ts` via `import type { PinoLike as Logger }`).

- **`IntentService.getOrCompute` (`modules/intent/service.ts`) treats "empty" and "stale" as different states — only empty (no row) triggers auto-recompute.** A stale row (`head_sha` != current PR head) is returned as-is on `GET /pulls/:id/intent`; only the manual `POST /pulls/:id/intent/recompute` (the `IntentCard` "Recompute" button) refreshes it. This is a deliberate cost-control choice — every new commit on a PR would otherwise silently trigger an LLM classification call on the next page load. If you touch this method, don't reintroduce a `headSha` staleness check into the auto path; add a UI staleness *indicator* instead if that's the actual ask.
- **`GET /pulls/:id/reviews` (`modules/reviews/routes.ts:129`) returns ALL reviews ever persisted for that PR, unfiltered — there is no `run_id` query param.** Any external consumer that starts a run via `POST /pulls/:id/review` (which returns `run_id` per `runs[]` entry, `modules/reviews/service.ts:108`) and then needs "the result of THIS run" must fetch this endpoint and filter client-side on `review.run_id === startedRun.run_id && review.kind === 'review'` — otherwise you'll pick up a stale/other-agent review from PR history. Built exactly this way for `@devdigest/mcp`'s `run-agent.service.ts` (`mcp/src/services/run-agent.service.ts`); reuse the pattern for any other out-of-process integration against this endpoint.
- **A `kind:'review'` row only appears in `GET /pulls/:id/reviews` once that agent's run has fully finished** — `runReview` (`modules/reviews/service.ts:103-138`) is fire-and-forget: it persists `agent_runs` rows and returns immediately, then executes in the background and writes the `reviews`/`findings` rows only on completion. So polling this endpoint right after `POST /pulls/:id/review` will correctly show nothing yet; absence is the expected "still running" state, not an error, until the matching `run_id` shows up.

## Tool & Library Notes

<!-- Dependency quirks, version gotchas, env/config oddities. -->
- **A Testcontainers `.it.test.ts` failing only in the FULL suite run (not in isolation) is very likely a flake, not a real regression — always re-run the full suite once before concluding a change broke something.** After extracting `resolveAttachedDocPaths` out of `run-executor.ts` (architecture-review fix), `run-executor-skills.it.test.ts` failed once in a full `npx vitest run` (`Cannot read properties of undefined (reading 'skills')` on `trace.prompt_assembly`) but passed cleanly both in isolation (`vitest run <file>`) and on a second full-suite run (252 passed, same as baseline). Root cause is contention between concurrently-spun-up Postgres Testcontainers under parallel test-file execution, not the code change. Before treating a suite failure as a regression from your diff, re-run the FULL suite (not just the one file) a second time — a true regression reproduces consistently; container-contention flakes usually don't.
- **`tsx watch` does not reliably hot-reload changes inside `src/vendor/`** — module-level constants computed at startup (e.g. `DEFAULTS` in `modules/settings/feature-models.ts`, computed from `FEATURE_MODELS`) stay stale after editing a vendor file until the server process is fully restarted. A page refresh is not enough. If a runtime behavior doesn't reflect a vendor-file change, restart the server manually (`Ctrl+C` → `pnpm dev`).
- **Killing the spawned `tsx watch` child process does NOT trigger a respawn** — `kill <pid>` on the actual `node .../src/server.ts` process (found via `lsof -i :3001`) leaves the parent `tsx watch` CLI process alive but idle; nothing relistens on the port. `tsx watch` only restarts on a file-change event, not on child exit. Fix: after killing the child, `touch src/server.ts` (or any watched file) to force the watcher to respawn it. Needed to pick up vendor-file edits when the dev server is running under `./scripts/dev.sh` in another terminal and you can't just re-run `pnpm dev` yourself.
- **`tsx watch` serves whatever branch is currently checked out on disk — `git checkout` to a DIFFERENT branch while the dev server is running silently changes/breaks its running routes, with no crash or warning.** Checked out a demo branch (`test/blast-radius-demo-pr`, forked from `main` before a feature module existed) to `git push` it, and the already-running dev server immediately started 404ing on that feature's routes — `tsx watch` picked up the file-tree change on disk mid-session and reloaded with the checked-out branch's code, not the branch the server was "started for". The fix was just switching back (`git checkout lesson-04`), but the failure mode looks exactly like a server crash/misconfiguration, not "you changed branches under a live server." When you need to `git checkout` a different branch for an unrelated task (pushing, diffing, cherry-picking) while `./scripts/dev.sh` is running in another terminal, expect the live app to reflect that checkout immediately — checkout back to the working branch as soon as the unrelated git op is done, and re-verify any endpoint you were testing.
- `@fastify/multipart` must be the **v10** major for Fastify v5 (v9 targets Fastify v4 and silently fails to register types). Register it once in `app.ts` next to cors/helmet with a `limits.fileSize` cap (2 MB for skill import). Gotcha: calling `req.file()` on a request that is NOT `multipart/form-data` (or has no file part) throws the plugin's own error that surfaces as **406**, not a clean 400. Wrap `await req.file()` in try/catch and throw `BadRequestError` so a missing/wrong upload is a 400 (see `modules/skills/routes.ts`, the `/skills/import/preview` handler).
- `BadRequestError` (400) did not exist in `platform/errors.ts` before the skills module — it was added there. Reuse it; don't reinvent a 400 in a module.
- **`MockLLMProvider`'s constructor `id` param (`server/src/adapters/mocks.ts:62`) is typed `'openai' | 'anthropic'` only** — narrower than `LLMProvider.id` (`'openai' | 'anthropic' | 'openrouter'`), so `new MockLLMProvider('openrouter', …)` fails to typecheck even though features like `review_intent` default to the `openrouter` provider. To mock an openrouter-backed feature in a test, construct with `'openai'` (the constructor id is cosmetic — it only sets `this.id`, unused by `completeStructured`) and register it under the real provider key: `overrides.llm = { openrouter: new MockLLMProvider('openai', { structured: fixture }) }`. `container.llm(provider)` looks up by the override map key, not the mock's internal `id`.
- **A zod field with `.default(0)` is still REQUIRED in the TS type produced by `z.infer<>`** — `.default()` only makes the field optional on `.parse()` *input*, not in the inferred output type used when a plain object literal is typed as `Agent` (e.g. test fixtures, hand-built DTOs). Adding `skill_count: z.number().int().default(0)` to the `Agent` contract broke two client test fixtures (`AgentCard.test.tsx`, `AgentEditor.test.tsx`) with "Property missing" until the field was added explicitly — TS didn't treat it as optional despite the runtime default.

## Tool & Library Notes (orchestration)
- **An `implementer` subagent's worktree edits are not guaranteed to be committed when it reports done, and its worktree branch may be forked from an old base (not the run's actual integration branch) if the branch is created fresh rather than off HEAD.** Running T1 of the project-context plan (`docs/plan/project-context-implementation-plan.md`), the implementer agent reported PASS with the right file diff, but `git merge <worktree-branch> --no-edit` on `lesson-05` returned "Already up to date" — its branch tip (`d584f14`) was an old ancestor of `lesson-05` (missing ~15 lesson-04/05 commits' worth of history), and the actual edits were still uncommitted in the worktree (`git status --short` showed them as modified, not `HEAD`). Fix applied: `cd` into the worktree, `git add`+`git commit` the edits there, then merge normally (a clean merge, since the stale base was still an ancestor of `lesson-05`). When orchestrating `implement-plan` waves: after an implementer reports PASS, always check `git -C <worktree-path> status --short` before merging — don't assume "PASS" implies "committed."

- **A new module's `service.ts` (Application layer) must route ALL filesystem/adapter access through a port, not just the reads that already have one.** `modules/project-context/service.ts` correctly used `GitClient.readFile` for content reads, but `discover()` still statically imported `walkMarkdownFiles` from `./walk.js` (raw `node:fs/promises` `readdir`/`stat`) and called `this.git.clonePathFor(ref)` to get a path for it — a real Application → concrete Infrastructure dependency that an architecture-reviewer pass caught as High severity. Fixed by adding `GitClient.listMarkdownFiles(repo)` to the port (`vendor/shared/adapters.ts`), implemented in `simple-git.ts` (which internally calls the same `walkMarkdownFiles` helper — the walk logic itself didn't move, only the call site did). Rule: when a service needs ANY filesystem/repo-clone access — not just content reads — check it's going through `GitClient` (or the relevant port), not a directly-imported fs helper, even if that helper is "just a pure function" in the same package.

- **A stale-base implementer worktree that ALSO touches `db/migrations/` can produce a numeric-index collision, not just a stale-history merge conflict.** During the PR Brief plan's T2 (add columns to `pr_brief` + generate a migration), the worktree branched before two unrelated, already-merged `lesson-05` migrations (`0013_futuristic_vindicator`, `0014_purple_wong` — pre-existing course content, nothing to do with this task) existed, so `pnpm db:generate` inside the worktree independently produced its own `0013_*.sql`/`0013_snapshot.json`, causing an `add/add` conflict on merge (not the "uncommitted edits" flavor documented above — this one has real, committed, genuinely conflicting migration-metadata files). Fix: merge the integration branch INTO the worktree branch first (not the other way around), then manually renumber the worktree's migration file/snapshot to the next free index, fix its snapshot's `prevId` to chain onto the now-latest snapshot's `id`, and append (not replace) its `_journal.json` entry — then merge the worktree branch back, which becomes a clean fast-forward. See the `_journal.json` `when`-timestamp gotcha above for the follow-on error this produced. Rule: any plan task that runs `db:generate` is a higher risk for this failure mode than a plain contract/route edit — check `ls server/src/db/migrations/*.sql | tail -3` on the integration branch right before dispatching such a task, and mention the expected next index in the task prompt.
- **The project-context discovery walk scans `GitClient.clonePathFor(repo)` (`server/clones/<owner>/<name>/`), a SEPARATE real git clone — not whatever's checked out in the live working directory.** Verified live: `specs/SPEC-01-project-context.md`, an untracked file in the actual dev-digest working directory (never committed), did NOT appear in the Project Context page's discovery list, while committed files like `.claude/skills/*`/`README.md` did. This is correct, expected behavior (AC-1 scans "the repo clone"), not a bug — but it means manually verifying this feature against your own uncommitted local edits will show nothing until those edits are committed and the app's clone is synced/refreshed. Also observed: after `run-executor`'s `checkoutPullHead` moved this clone to a PR's head commit, the NEXT page load's discovery count changed (125 → 195 files) because the walk always reflects whatever commit is currently checked out — confirms the documented "no persisted timestamp, always a live scan" behavior (AC-4a) concretely, including the side effect that a review run can change what the next discovery scan sees.

- **A stale-base implementer worktree missing an earlier task's new export will sometimes "solve" it by locally redeclaring a structurally-compatible placeholder instead of reporting itself blocked — this must be manually swapped for the real import at merge time, or the codebase silently carries two divergent definitions of the same type.** T5 of the PR Brief plan needed `RiskLevel`/`ReviewFocusItem` from T1's contract extension, but its worktree was branched before T1 was merged; rather than flagging "blocked, T1's export not found," the implementer wrote local `export const RiskLevel = z.enum([...])` / `ReviewFocusItem = z.object({...})` copies with a large in-file comment explaining the workaround and instructions to delete them later. This typechecked fine in isolation and even matched the real shapes correctly, but if merged as-is it would leave two independent enum/object definitions doing the same job with no static link between them. Caught by re-reading the file before merging (not by any automated check). Rule: when a task's report mentions redeclaring/duplicating a type "because the real one wasn't visible," always diff that declaration against the actual merged source before merging, and manually swap it for a real import — don't trust that a locally-typechecking placeholder is equivalent to the genuine cross-module contract.

- **When a stale-base worktree is so far behind that an implementer's OWN sibling-module dependencies (not just unrelated history) are missing, the correct agent behavior is to report itself blocked with evidence and refuse to fabricate — and simply re-dispatching the same task (a fresh `Agent` call, new worktree) is a valid, usually-successful fix**, not a sign the task itself is broken. T8 of the PR Brief plan (depends on T3-T7, all already merged) got a worktree with zero brief-module files and the old `PrBrief` shape; it correctly refused to guess at `input-builder.ts`/`brief-prompt.ts`/`validate.ts`/`repository.ts`'s exported signatures and stopped with a full evidence trail (git log, `ls`, grep results) instead of inventing stand-ins. Re-dispatching the identical task prompt (plus an explicit "verify via `git log`/`ls`/`grep` BEFORE writing any code, stop immediately if missing" first step) got a fresh worktree draw that had the real dependency chain. Don't add retry/backoff logic to the orchestration process for this — a single manual re-dispatch is cheap and the failure is self-diagnosing from the agent's own report.

- **(2026-07-12 root-cause fix for the recurring "stale worktree base" pattern above)** All of it traced to `worktree.baseRef` defaulting to `"fresh"` (branch from `origin/<default-branch>`), while this project's actual integration branch (`lesson-05`) is local-only and never pushed — so every implementer worktree branched from `origin/main`'s stale tip, missing all of the current run's already-merged work, no matter how recently it merged. Confirmed directly: `git log --oneline origin/main -3` matched every "stale" worktree's HEAD exactly. Fix: set `"worktree": {"baseRef": "head"}` in `.claude/settings.local.json` — new worktrees then branch from the current local HEAD instead. Immediately fixed a task (T8 of the PR Brief plan) that had failed twice in a row purely from staleness, with zero further merge conflicts or blocked-reports afterward. **When running `/implement-plan` (or any implementer-worktree orchestration) against a local-only integration branch that isn't pushed to `origin`, set this config BEFORE dispatching the first task**, not after repeated failures — the earlier T1-T7 workaround (manually `git merge lesson-05` into each worktree before merging back) still works but is pure waste once this setting is corrected.

## Recurring Errors & Fixes
<!-- Error signature → root cause → fix. -->
- **`error TS1160: Unterminated template literal` pointing at the LAST line of a file with no template literals in it** → root cause: a `/** ... */` doc comment contained a literal glob-style path with a `*/` substring inside it (e.g. writing the path `modules` + `/*` + `/routes.ts` back-to-back in prose) — the `*/` closes the JSDoc comment early, so the rest of the comment text (including its closing backtick) gets scanned as real code, and the orphaned backtick is parsed as an unterminated template literal running to EOF. The error location (end of file) is nowhere near the actual typo, which makes it look like a stale incremental-build/cache issue instead — `npx tsc --noEmit <that one file>` reproduces it in isolation and rules that out in seconds. Fix: never let the two characters `*` immediately followed by `/` appear inside a `/** */` block comment's text (rephrase glob-like paths to avoid it, e.g. "module's `routes.ts` file" instead of a literal `*/`-containing glob).
- **Adding a required field to a vendored zod contract (e.g. `status` on `BlastRadius`) breaks literal object-construction fixtures OUTSIDE `src/`, not just inside it.** Grepping only `server/src` for `BlastRadius\b` before the edit found zero construction sites, but `server/test/contracts.test.ts` (a top-level `test/` fixture, sibling to `src/`, not inside it) built a literal `BlastRadius.parse({...})` without `status` and failed at `pnpm test` time — typecheck alone didn't catch it because `.parse()` fixtures are plain object literals, not statically typed as the contract. When adding a required field to any `@devdigest/shared` contract, grep the WHOLE package (`server/`, not `server/src/`) for the type name, including `test/*.test.ts`, before declaring "no construction sites found."
- **`PostgresError: column "X" of relation "Y" already exists` from `pnpm db:migrate`, even though the migration's SQL hash is already present in `__drizzle_migrations`** → root cause: `drizzle-orm/postgres-js/migrator`'s `migrate()` does not decide "already applied" purely by hash-set membership — it also compares each local migration's `_journal.json` `when` (ms timestamp) against the last-applied row's `created_at`, and treats any local entry with a `when` greater than that as pending regardless of whether an identical hash already exists in the table. Hit this after manually renumbering a colliding migration file (see the orchestration note below): the new `_journal.json` entry was given a fresh `when` (current time) instead of the original one, which made the migrator re-run already-applied `ALTER TABLE ... ADD COLUMN` statements. Fix: when hand-editing `_journal.json` to fix an index/tag after a rename, copy the **original** `when` value the migration was actually applied with (find it via `SELECT created_at FROM drizzle.__drizzle_migrations ORDER BY id` and match by row position/hash) — do not regenerate a new timestamp.
- **`PostgresError: invalid input syntax for type uuid: "system"` inserting into `repos`** → `repos.createdBy` (`server/src/db/schema/repos.ts:18`) is `uuid('created_by').references(() => users.id)`, but `conventions.it.test.ts` and `conventions/repository.test.ts` insert `createdBy: 'system'` (a leftover string sentinel from before the column was a real FK). Both files fail as of 2026-07-02 on this branch — pre-existing, unrelated to the Intent Layer work. Fix: omit `createdBy` in test fixtures (it's nullable) or insert a real seeded user id; do not pass a non-uuid string.
- **Cost shown is ~2× actual OpenRouter cost** → pricing table in `server/src/adapters/llm/pricing.ts` has wrong values. Root cause: the table is the fallback when `usage.cost` from the API is absent, so any table error shows up directly. Fix: verify each model's prices against `openrouter.ai/models` before adding. Example: `deepseek/deepseek-v4-flash` was `{ in: 0.14, out: 0.28 }` (2× too high); corrected to `{ in: 0.07, out: 0.14 }`.
- **Any feature that defaults to the `openrouter` provider and calls `llm.complete(...)` (plain, unstructured) silently returns an empty/degraded result on every real request — and a Docker-gated integration test using `MockLLMProvider` will NOT catch it.** Root cause: the shared `OpenRouterProvider` (`reviewer-core/src/llm/openrouter.ts:158`, "the one OpenAI-compatible **structured** provider" per `reviewer-core/CLAUDE.md`) only implements `completeStructured` — its `complete()` always `throw`s `NOT_SUPPORTED`. Blast's `explain()` called `llm.complete(...)`, whose own try/catch caught that throw and gracefully degraded to `{summary:''}` — so it LOOKED like a working feature returning an empty answer (or a missing-API-key issue, which is what I first assumed) rather than a crash. The integration test still passed because `MockLLMProvider.complete()` (`adapters/mocks.ts`) is implemented generically and doesn't share the real provider's constraint — mocks that are more capable than the real adapter they stand in for will hide exactly this class of bug. Fix: `completeStructured` with a single-field schema (`z.object({ summary: z.string() })`) instead of `complete`, mirroring every other LLM-backed feature in this codebase (`classifyIntent`, `extractConventions` — grep confirmed **zero** other `.complete(` call sites in `server/src` before this). Rule: before writing a new plain-prose LLM call, grep the codebase for existing `.complete(` usages — if there are none, that itself is a signal the default provider doesn't support it. And when a live end-to-end check (curl/browser) shows an empty/degraded LLM result, verify the real provider adapter directly (not just re-run the mocked test) before concluding it's a config/env problem.
