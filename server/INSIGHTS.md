## What Works
<!-- Approaches/solutions that worked. e.g. "X via Y in src/foo.ts:42 because …" -->
- Computing PR-list cost with a separate `SELECT cost_usd FROM agent_runs WHERE status='done' ORDER BY ran_at DESC` + JS grouping (same pattern as score in `modules/pulls/routes.ts:119-130`) — cheap enough for a small PR list, keeps the query readable, and avoids a complex JOIN.
- Spreading `{ usage: { include: true } }` into the OpenAI SDK `chat.completions.create()` call DOES reach OpenRouter — the SDK passes untyped extra body fields through without stripping them. Confirmed: `res.usage.cost` is populated for both plain completions and structured output (`response_format: json_schema`) calls. The cast `(res.usage as { cost?: number })?.cost` is correct.

## What Doesn't Work
<!-- Dead ends & antipatterns. The most valuable & most-skipped section. -->
- **Returning raw Drizzle camelCase rows from a new module's `service.ts`/`routes.ts` while the client types the response as a snake_case Zod contract (`EvalCase`) passes both packages' `tsc` and breaks at runtime.** `api.get<EvalCase[]>` is an unchecked generic cast, and several contract fields are `z.unknown()`, so `c.input_diff` / `c.name` read `undefined` off `{ inputDiff, … }` with no compile error. Caught by plan-verifier on SPEC-03 (T10/T14) after implementers green-lit typecheck. Fix: map every service exit through a single mapper (`toEvalCaseDto` in `modules/eval/service.ts:65`) — same idea as `toAgentDto` in agents — before the route returns. Rule: for any new module that exposes a vendored snake_case contract, grep the service return paths for `$inferSelect` / raw table rows and confirm a DTO mapper sits between repository and HTTP.
- **`polling/routes.ts` `onConflictDoUpdate` only updates what you explicitly list — diff-size fields were missing and silently stayed stale.** The original set included `title`, `headSha`, `status`, `updatedAt` but omitted `additions`, `deletions`, `filesCount`. Those three are written on initial INSERT (new PRs) but never refreshed, so the SIZE column showed permanently stale values even after a poll. Fix: add them explicitly to the conflict set (`polling/routes.ts:50-60`). Rule: whenever adding a new column to `pull_requests`, check whether it belongs in the conflict-update set too.
- **Changing an EXISTING `FEATURE_MODELS` entry's default (not just adding a new one) can silently break an integration test fixture that hardcoded the old default, and `tsc` will not catch it — only running the full test suite will.** T3 of the PR Brief plan flipped `risk_brief`'s default from `openai`/`gpt-4.1` to `openrouter`/`deepseek-v4-flash`; `test/settings-models.it.test.ts` had an unrelated assertion (`resolveFeatureModel(..., 'risk_brief')` used as a generic "still resolves to registry default" example) hardcoding the pre-T3 values, and it stayed green through T3-T17's individual per-task test runs because none of them happened to touch that specific test file. Only surfaced when re-running the FULL `npx vitest run` after later unrelated fixes. Rule: after changing any existing `FEATURE_MODELS` entry's default provider/model, grep the whole `server/` tree (test files included) for that feature id string, not just the files you're editing — a stale expected-default assertion elsewhere is a real, silent regression risk that typecheck cannot catch (it's a runtime string comparison, not a type).
- **New `FEATURE_MODELS` entries must default to `openrouter`, not `openai`** — the `conventions` entry shipped with `defaultProvider: 'openai'`/`defaultModel: 'gpt-5.4'`, which silently demands `OPENAI_API_KEY` at runtime even when the workspace has only `OPENROUTER_API_KEY` configured. Every other feature (`onboarding`, etc.) defaults to `openrouter`. Check `FEATURE_MODELS` in both vendor copies of `platform.ts` (`server/src/vendor/shared/contracts/platform.ts` and `client/src/vendor/shared/contracts/platform.ts`) when adding a new feature — the default propagates directly to `resolveFeatureModel` → `container.llm(provider)`.
- "Take the single newest completed `agent_run` for PR-list cost" is wrong when multiple agents run together — you get only one agent's cost. The correct definition is **sum of the latest run per agent** (newest `ran_at` per `agent_id`, then sum `cost_usd`). Implemented in `modules/pulls/routes.ts` via a `seenAgentPerPr` Set + accumulator loop over rows sorted `ran_at DESC`.
- `run-executor.ts` was calling `priceBook.estimate()` unconditionally, discarding `outcome.costUsd` that the engine already computed from the OpenRouter API. The engine (`reviewPullRequest` in `reviewer-core`) returns `costUsd` on the `ReviewOutcome` — always prefer it; fall back to estimate only when null: `outcome.costUsd ?? priceBook.estimate(agent.model, tokensIn, tokensOut)` (fixed in `modules/reviews/run-executor.ts:214`).
- **(2026-06-19 correction of above)** "Sum of latest run per agent" is also wrong — it mixes agents from different review sessions (e.g. Security from yesterday + General from today). The correct approach is a **60-second session window**: find the newest `ran_at` for the PR, then sum `cost_usd` of all completed runs within 60 s of it. Agents launched together by one "Run Review" click start within milliseconds; 60 s safely covers any DB round-trip lag. Implemented in `modules/pulls/routes.ts` (two-pass: find `latestRanAtByPr`, then filter + sum).
- **Persisting only the RATIO of a metric and not its raw counts silently forces every later aggregate to be a wrong one — an unweighted mean of means.** SPEC-03's `eval_runs` stored `recall`/`precision`/`citation_accuracy` as scalars but discarded the counts they came from, so `aggregateRunRows` (`modules/eval/scorer.ts`) could only *average per-case ratios* to build the dashboard. That gives a 1-expectation case the same weight as a 10-expectation one, which is not what the spec's "across the cases in the run" means. Worse, it is not fixable after the fact: `citation_accuracy = kept / (kept + dropped)`, and **`dropped` is recoverable from nowhere** — `actual_output` holds only the findings that survived the grounding gate, so the denominator is simply gone. Fix (migration `0017`): store `matched`, `expected_total`, `produced`, `false_positives`, `kept`, `dropped` on `eval_runs` and POOL them, through the *same* `poolMetrics()` the live scorer uses, so the number shown right after a run and the number on the dashboard cannot drift. **Rule: when a run-level metric is a ratio, persist the numerator and denominator, not the quotient** — a quotient cannot be re-aggregated, and you usually discover this only when someone asks why the dashboard disagrees with the run.
- **A derived "did it pass?" is not the same as the scorer's verdict, and re-deriving it from the metrics inverts the result for the empty-expectation case.** `EvalCaseModal` computed `passed = recall > 0 || precision > 0`. But an eval case whose `expected_output` is `[]` (the dismissed-finding shape, "flag nothing here") scores **recall = 1 by definition** — the empty-set convention. So a run that *failed* by producing findings it should not have still rendered as **passed**, on the primary user story's final step. The scorer already emits the authoritative `pass` (`scoreCase`, AC-30) and it rides along on `per_trace[].pass`. **Rule: if a pure scorer already decides pass/fail, never recompute that decision downstream from its outputs** — the empty-set conventions (recall = 1, precision = 1 when there is nothing to be wrong about) make the metrics deliberately flattering in exactly the cases the pass flag is there to catch.

- **`matchesExpectation` (`modules/eval/scorer.ts:35`) line-range matching was an exact intersection, causing real false negatives when the agent's cited line was 1-off from the eval case's expected line** — e.g. case `truthiness-trap-drops-valid-falsy-settings-values` expected line 53, agent produced line 52, recall scored 0% despite the finding being semantically correct. Fixed by widening only the `produced` (actual agent output) range by ±`LINE_TOLERANCE` (3) lines before the intersection check — the `expected` range from the eval case is deliberately left untouched, since tolerance belongs to "how precisely can the agent cite a line," not to loosening the eval case's own definition. Keep this asymmetry if extending tolerance further: never widen `expected`.

- **Changing an agent's linked-skill set (and its `contextDocs`) did NOT bump `agents.version`, even though both materially change every review/eval run — so the eval "version run" model silently collapsed skills-only experiments into one bucket.** Two paths bypass versioning: (a) `isConfigChange` (`modules/agents/helpers.ts:65`) only lists name/description/provider/model/systemPrompt/strategy/ciFailOn/repoIntel/outputSchema — **not** skills, **not** contextDocs; (b) skill links are written through `setSkills`/`linkSkill` (`modules/agents/repository.ts`), which touch only the `agent_skills` table and never call `agents.update`, so no version bump and no `agent_versions` snapshot ever happens for a skill edit. The runner injects linked+enabled skill bodies live (`modules/eval/runner.ts:46`), so an eval run before and after a skills change produces genuinely different output — but both rows carry the same `agent_version`, and SPEC-04's "latest run per case within one agent version" dedup keeps only the newest, so the dashboard shows "no difference between runs" and Compare has nothing to compare. Fix (2026-07-15): added `AgentsRepository.bumpVersion(workspaceId, id)` (increment + `snapshotVersion`, which already records skill ids) and called it from `AgentsService.setSkills`/`linkSkill` **only on a real, order-sensitive change** (`sameOrderedIds` in `helpers.ts` — reordering also changes the assembled prompt, so position matters; a no-op save must not churn the version). **`contextDocs` still has this exact bug** — `setContextDocs` routes through `repo.update({ contextDocs })` but `isConfigChange` ignores `contextDocs`, so context-doc edits remain unversioned; left as-is (out of scope) but flagged. Rule: anything the runner reads at eval/review time (skills, context docs, prompt, model, strategy) must bump the agent version, or the version-run dashboard cannot represent a change to it.
- **`github.event.pull_request.head.repo.fork == false` is the wrong AC-19 guard when the *installed* repo is itself a GitHub fork — it skips every same-repo branch PR with no error.** `head.repo.fork` is `true` for all PR heads inside `RomanMinenok/dev-digest` (forked from `burnjohn/dev-digest`), including internal PRs like `test/ci-run → devdigest/ci`; the job shows `skipped` immediately. What we actually need is "head branch lives in this repository, not an external contributor fork": `head.repo.full_name == github.repository`. External fork PRs are still blocked (GitHub withholds repo secrets on `pull_request` from forks); same-repo branches in a downstream fork *do* receive secrets and are safe to run because the workflow executes trusted base-repo YAML, not attacker-submitted workflow from the fork head. Fixed in `modules/ci/workflow.ts:46` and mirrored in `validate-workflow.ts:19` (`SAME_REPO_CONDITION_PATTERN`) so manual overrides cannot weaken the guard.

## Codebase Patterns
<!-- Conventions & architectural decisions, with the "why". -->
- **When emitting Markdown that wraps arbitrary agent/model text (system prompts, LLM output), a fixed ```` ``` ```` fence silently breaks the document — those strings routinely contain their own ```` ``` ```` code fences.** The eval run-log (`modules/eval/run-log.ts`, `.md` output) wraps the assembled system/user prompt in a fence whose length is computed per-block: `Math.max(3, longestBacktickRunInside + 1)` backticks (`fenced()` helper). A system prompt containing a ```` ```js ```` example then gets wrapped in a 4-backtick fence and renders correctly. Reuse this `fenced()` approach for any future feature that dumps prompt/model text into Markdown — never hardcode a 3-backtick fence around untrusted-length content.
- **Eval capture must never re-feed `trace.prompt_assembly.intent` into a case's frozen enrichment — that string is already `wrapUntrusted`-wrapped (`reviewer-core/src/prompt.ts:176`), so replaying it through `assemblePrompt` double-wraps the intent block.** Source the structured intent via `container.intentRepo.get(pr_id)` instead, and take `callers`/`repo_map` from `prompt_assembly` raw fields plus `context_docs` from `RunTrace.specs_read` (dropping `content: null`). Implemented in `modules/eval/capture.ts` + `EvalService.captureEnrichment` (`modules/eval/service.ts`). Same class of bug: recovering `rank_note` by stripping it off `prompt_assembly.user` — recompute from `repoIntel.getFileRank` + shared `rankNoteSentence` (`reviews/helpers.ts`) instead.
- **`SEED_SKILLS` (`db/seed-skills.ts`) and `agentSkillLinks` (`db/seed.ts`) are two independent lists with no cross-check** — nothing enforces that every seeded agent gets a linked skill, or that the skill count meets a course-lesson minimum. Found via review feedback: only 4 skills existed (course required ≥6) and 3 of the 5 seed agents (`General Reviewer`, `Security Reviewer`, `Performance Reviewer`) had zero entries in `agentSkillLinks` despite their `description` implying skill-driven behavior. Fixed by adding `security-rubric`/`performance-rubric` to `SEED_SKILLS` and a link per previously-unlinked agent. When adding a new seed agent or skill, manually verify both lists — there is no test asserting "every agent has ≥1 linked skill" or a minimum skill count.
- `ReviewRepository` in `modules/reviews/repository.ts` is a class wrapper around the function-level repos in `repository/run.repo.ts`, `repository/review.repo.ts`, etc. When adding a field to `completeAgentRun`, you must update **both** the underlying function signature in `run.repo.ts` AND the class method signature in `repository.ts` — a mismatch causes a TS error at the `run-executor.ts` call site, not at the definition, which is confusing.
- **Linked skills reach a review run ONLY through `run-executor.ts`, not the engine or the agents module.** The engine (`reviewPullRequest`) and `assemblePrompt` already accept a resolved `skills: string[]`, but until `run-executor.ts` loads them, linking a skill to an agent has zero effect on a review. The executor loads them via its constructor-injected agents repo (`this.agents.linkedSkills(agent.id)` = `container.agentsRepo`, already ordered by `agent_skills.order`), filters to `skill.enabled`, and passes `...(skillBodies.length ? { skills } : {})` (omit-when-empty, like `callers`/`repoMap`). The in-prompt gate is therefore **linked AND `enabled`**, enforced at that filter — not in the engine. Skill bodies are passed as trusted instructions (NOT `wrapUntrusted`); the trust mitigation is import-disabled-by-default + manual vetting.
- `agent_runs` has **no direct FK to `findings`**. The join chain for per-run severity counts is `findings.review_id → reviews.id → reviews.run_id`. To query severity breakdown per run: `JOIN reviews ON findings.review_id = reviews.id WHERE reviews.run_id IN (...)` grouped by `reviews.run_id, findings.severity`. There is no shortcut via `agent_runs` — the link only exists through `reviews`. See `modules/reviews/repository/run.repo.ts:listRunsForPull` and `modules/pulls/routes.ts` for the implemented pattern.
- **`@devdigest/shared` is vendored in two places with no automated sync.** `server/src/vendor/shared/` is the canonical copy; `client/src/vendor/shared/` is a manual mirror. TypeScript will NOT catch drift between them — each package resolves `@devdigest/shared` to its own local copy, so both sides compile independently even when the schemas diverge. Any edit to a contract file must be applied to both directories. The breach patterns and detection commands for drift are in `docs/skills/contract_breach.md`.
- **Anti-hallucination gate for LLM extraction**: after `completeStructured`, verify each candidate by taking `evidence_snippet.split('\n')[0].trim()` and checking the referenced file on disk literally `includes()` that string. Drop the candidate if the file is missing or the first line is absent. This kills hallucinated `file:line` references without fuzzy matching or embeddings. Implemented in `modules/conventions/extractor.ts`; re-use this pattern for any future LLM feature that returns evidence paths.
- **`repoIntel.getConventionSamples()` returns `[]` when the index is disabled or not yet built** (the `RepoIntel` degraded contract). LLM features that need sample source files must have a fallback or the scan silently operates on zero files. Convention extractor handles this with `fallbackWalk` in `modules/conventions/helpers.ts` — a simple BFS over the clone filtered to `.ts/.tsx/.js/.jsx`, excluding `node_modules`/`dist`/`.git`/migrations/test files. Apply the same pattern for future features that consume `getConventionSamples` or similar `RepoIntel` methods.
- **The persisted `references` table (repo-intel's caller graph, `getResolvedCallers` in `modules/repo-intel/repository.ts:503`) does NOT resolve calls made through an interface-typed property back to the concrete implementing class's method.** `LocalNoAuthProvider.currentUser`/`currentWorkspace` (`adapters/auth/local.ts`) are called ~30 times as `container.auth.currentUser(req)` (`container.auth` is typed `AuthProvider`), yet a direct query — `SELECT count(*) FROM "references" WHERE decl_file = 'server/src/adapters/auth/local.ts' AND to_symbol = 'currentUser'` — returns **0**. The indexer's reference resolution matches call sites to declarations by name+file heuristics that don't cross an interface/DI boundary, so any method only ever invoked via `container.<interface>.method(...)` will show as having 0 callers in Blast Radius even though it's very much alive. Don't read "0 callers" in Blast Radius as "dead code" for anything reached through an injected interface (`container.auth`, `container.repoIntel`, `container.llm`, etc.) — check real call sites with a plain grep before concluding a symbol is unused.
- **`GitClient.readFile(repo, path)` reads the clone's CURRENT working-tree checkout — NOT a specific sha; there is no head-pinned read API** (`@devdigest/shared` `adapters.ts` `GitClient`). To read a repo file "at the PR head" you must sync the clone to that head first — only `modules/repo-intel/service.ts:151` does an explicit `git.sync`; the review diff path (`loadDiff`) just uses whatever is checked out and falls back to persisted `pr_files` patches. So `modules/intent/spec-resolver.ts` (wired at `modules/intent/service.ts:84`) reads repo `.md` from the current checkout, which is not guaranteed to be the PR head. If you write a feature that must read repo files at a precise commit, sync first — don't trust a `readFile(sha, …)` signature (it doesn't exist; plan prose that says so is wrong).
- **Unit-testing a service method that mixes DB reads with pure composition logic: extract the pure part into an exported function taking plain data, rather than mocking Drizzle's chainable query builder.** `PullsService.smartDiff` needed tests for session-window filtering, classification grouping, and split-threshold logic (Decision 6 in the Smart Diff plan) but also does two `container.db.select().from().where()` calls and instantiates `ReviewService`. Mocking that chain (or `ReviewService`) for a plain unit test is awkward and brittle. Fix: extracted `assembleSmartDiff(prFiles, reviews)` as a pure function in `modules/pulls/service.ts` — `smartDiff()` just fetches and delegates to it. Tests call `assembleSmartDiff` directly with in-memory fixtures, no mocking needed. Reuse this split (thin DB-fetching method + pure composition function) for any future service method that needs unit tests without Testcontainers.
- **`pulls/service.ts` is the only `service.ts` in the codebase that talks to Drizzle directly (`import { and, eq } from 'drizzle-orm'` + `container.db.select().from(t.pullRequests)...`) instead of going through a `repository.ts`.** Every other module with a service (`reviews`, `agents`, `conventions`, `intent`, `repo-intel`, `repos`, `skills`) routes DB access through a repository. This is a deliberate, scoped exception (Smart Diff plan Decision 1 — "repository.ts is not added ... matching the module's current pre-existing, non-canonical style rather than introducing a disproportionate refactor"), confirmed by an architecture review as sound-with-risk (Medium severity) rather than a defect. Don't copy `pulls/service.ts`'s direct-Drizzle style as a template for a new module's `service.ts` — extract a `repository.ts` instead, matching the other 7.
- **`PinoLike` (`platform/run-logger.ts:21`) is the canonical minimal structural logger type** (`(obj, msg?)`, pino-compatible). `modules/reviews/run-executor.ts:21` has an identical duplicate exported as `Logger`. A new module that needs a logger param should import `PinoLike` from `platform/run-logger`, NOT `Logger` from `modules/reviews/run-executor` — the latter creates a wrong-direction `module → reviews` import edge (flagged in the intent architecture review; fixed at `modules/intent/service.ts` via `import type { PinoLike as Logger }`).

- **`IntentService.getOrCompute` (`modules/intent/service.ts`) treats "empty" and "stale" as different states — only empty (no row) triggers auto-recompute.** A stale row (`head_sha` != current PR head) is returned as-is on `GET /pulls/:id/intent`; only the manual `POST /pulls/:id/intent/recompute` (the `IntentCard` "Recompute" button) refreshes it. This is a deliberate cost-control choice — every new commit on a PR would otherwise silently trigger an LLM classification call on the next page load. If you touch this method, don't reintroduce a `headSha` staleness check into the auto path; add a UI staleness *indicator* instead if that's the actual ask.
- **`GET /pulls/:id/reviews` (`modules/reviews/routes.ts:129`) returns ALL reviews ever persisted for that PR, unfiltered — there is no `run_id` query param.** Any external consumer that starts a run via `POST /pulls/:id/review` (which returns `run_id` per `runs[]` entry, `modules/reviews/service.ts:108`) and then needs "the result of THIS run" must fetch this endpoint and filter client-side on `review.run_id === startedRun.run_id && review.kind === 'review'` — otherwise you'll pick up a stale/other-agent review from PR history. Built exactly this way for `@devdigest/mcp`'s `run-agent.service.ts` (`mcp/src/services/run-agent.service.ts`); reuse the pattern for any other out-of-process integration against this endpoint.
- **A `kind:'review'` row only appears in `GET /pulls/:id/reviews` once that agent's run has fully finished** — `runReview` (`modules/reviews/service.ts:103-138`) is fire-and-forget: it persists `agent_runs` rows and returns immediately, then executes in the background and writes the `reviews`/`findings` rows only on completion. So polling this endpoint right after `POST /pulls/:id/review` will correctly show nothing yet; absence is the expected "still running" state, not an error, until the matching `run_id` shows up.
- **Two incompatible conventions exist for local debug-log flags: `CONVENTIONS_DEBUG` (`modules/conventions/extractor.ts:8`) reads `process.env.X === 'true'` directly at module scope, while `EVAL_LOG_PROMPT` (added for eval-run prompt logging) goes through the zod-validated `AppConfig`/`EnvSchema` in `platform/config.ts` (same pattern as `EMBEDDINGS_ENABLED`/`REPO_INTEL_ENABLED`) and is read via `container.config.evalPromptLogEnabled`.** The module header comment on `config.ts` explains why *secrets* must never be read via raw `process.env` (only `SecretsProvider`), but says nothing about plain debug toggles — so there's no documented rule for which pattern a new one should follow. Prefer the `AppConfig` route for anything reachable through `container` (testable via `ContainerOverrides`, visible in one schema); a raw `process.env.X` module constant only makes sense for a flag scoped to a single file with no DI path to it.
- **`server/tsconfig.json` sets neither `noUnusedLocals` nor `noUnusedParameters`, so deleting a tenancy/authz check leaves `pnpm typecheck` fully green with no signal that scoping was removed.** Confirmed while deliberately building 10 security-bug fixtures for a review-eval PR: dropping the `eq(t.pullRequests.workspaceId, workspaceId)` clause from `pulls/repository.ts::findPr`, the `eq(t.agentRuns.workspaceId, workspaceId)` clause from `reviews/repository/run.repo.ts::deleteAgentRun`, and the `getById(workspaceId, agentId)` ownership check from `agents/service.ts::listVersions` (leaving `workspaceId` as an unused parameter each time) all typecheck cleanly. Don't treat "typecheck passed" as evidence that workspace/tenant scoping survived a refactor — diff for `workspaceId`/`tenantId` disappearing from a `.where(...)` clause or an early-return ownership guard instead.
- **Cross-module writes that must share one Drizzle transaction cannot go through a class wrapper that binds `this.db` — inject a tx-aware port from the composition root instead.** SPEC-05's `CiRepository.ingestRun` needs one atomic transaction for `agent_runs` + `run_traces` + `ci_runs`. `ReviewRepository.saveRunTrace` (`modules/reviews/repository.ts`) always uses `this.db`, so it cannot join `CiRepository`'s `db.transaction(...)`. An architecture-reviewer High finding also forbade a direct `ci → reviews/run.repo` import. Fix: keep `createCiAgentRun` / `saveRunTrace` as function-level helpers in `modules/reviews/repository/run.repo.ts:174,250` (they already take `db: Db`, so a tx handle works), define a narrow `CiRunPersistence` port on `CiRepository` (`modules/ci/repository.ts:7`), and wire the real functions only in `platform/container.ts:33,160`. Rule: when module A must write module B's tables inside A's transaction, B exposes tx-accepting functions and the composition root injects them — never import B's repository from A, and never call a `this.db`-bound wrapper from inside a foreign transaction.
- **(2026-07-16 correction of the `CiRunPersistence` entry above)** Passing `(db: Db, values) => …` on the port still leaked Drizzle into the module surface (architecture Medium). Prefer a **Db-free** writer type — `PersistCiIngestRun = (input: MappedCiIngestInput) => Promise<CiRunRow>` on `modules/ci/repository.ts` — whose implementation factory lives under `platform/` (`ci-ingest-persist.ts`), owns `db.transaction`, and is the only file that imports both `ci` types and `reviews/run.repo`. Map artifact→insert fields in the application layer (`modules/ci/map-ingest.ts`) before calling `ingestRun`; the repository/persister must not resolve `provider ?? trace.config.provider ?? agent.provider` itself. The earlier "inject createCiAgentRun/saveRunTrace taking `Db`" shape is obsolete for this module.

## Tool & Library Notes

<!-- Dependency quirks, version gotchas, env/config oddities. -->
- **A Drizzle migration snapshot can be missing columns that earlier migrations already added — `pnpm db:generate` then emits a wrong next migration (or wants to re-add/drop those columns), not just a numeric index collision.** Hit on SPEC-03 T2: tip was correctly `0015_*.sql`, but `meta/0015_snapshot.json` had lost `context_docs` on `skills`/`agents` (columns from 0013/0014, wiped by a prior stale-worktree generate). First generate of 0016 therefore wasn't a clean one-column `agent_version` ADD. Fix: restore the missing columns into the tip snapshot so it matches live DB + prior migration SQL, delete the bad `0016_*`, regenerate. Distinct from the add/add index collision (see orchestration notes below) — here the *filename* index was right and the *snapshot contents* were wrong. Before trusting a fresh `001N_*.sql`, diff it against the schema change you intended; if it mentions unrelated tables/columns, inspect `meta/001(N-1)_snapshot.json` first.
- **A Testcontainers `.it.test.ts` failing only in the FULL suite run (not in isolation) is very likely a flake, not a real regression — always re-run the full suite once before concluding a change broke something.** After extracting `resolveAttachedDocPaths` out of `run-executor.ts` (architecture-review fix), `run-executor-skills.it.test.ts` failed once in a full `npx vitest run` (`Cannot read properties of undefined (reading 'skills')` on `trace.prompt_assembly`) but passed cleanly both in isolation (`vitest run <file>`) and on a second full-suite run (252 passed, same as baseline). Root cause is contention between concurrently-spun-up Postgres Testcontainers under parallel test-file execution, not the code change. Before treating a suite failure as a regression from your diff, re-run the FULL suite (not just the one file) a second time — a true regression reproduces consistently; container-contention flakes usually don't.
- **`tsx watch` does not reliably hot-reload changes inside `src/vendor/`** — module-level constants computed at startup (e.g. `DEFAULTS` in `modules/settings/feature-models.ts`, computed from `FEATURE_MODELS`) stay stale after editing a vendor file until the server process is fully restarted. A page refresh is not enough. If a runtime behavior doesn't reflect a vendor-file change, restart the server manually (`Ctrl+C` → `pnpm dev`).

- **(2026-07-12 root-cause fix for the recurring "stale worktree base" pattern above)** All of it traced to `worktree.baseRef` defaulting to `"fresh"` (branch from `origin/<default-branch>`), while this project's actual integration branch (`lesson-05`) is local-only and never pushed — so every implementer worktree branched from `origin/main`'s stale tip, missing all of the current run's already-merged work, no matter how recently it merged. - **(architecture-review finding, 2026-07-12) The 60-second "review session" window is now defined independently in three places, not one shared rule.** `pulls/service.ts`'s `assembleSmartDiff` computes it as a pure application-layer function over `reviews.created_at`; `brief/repository.ts:28-33` redefines the same `SESSION_WINDOW_MS = 60_000` constant but applies it in the **repository** (infrastructure layer) over `agent_runs.ran_at`/`status='done'`; and the client's `PrBriefCard.tsx` re-derives a looser proxy ("any review exists" / `reviews[0]`) instead of consuming a server-provided flag. - **CI Runs shows a "Failed"/error row with no PR, cost, findings, or trace after a review successfully posted "Changes requested" on GitHub** → root cause: GitHub Actions skips later steps after `Run DevDigest review` exits non-zero for blocking findings, so `Upload review result` never runs unless it has `if: always()`. The runner writes `devdigest-result-<slug>.json` before returning exit code 1 (`agent-runner/src/run.ts:161-214`), but the workflow must still upload it (`modules/ci/workflow.ts:58-65`) or ingest only sees a completed Actions run with no artifact and persists a status-only error row. Fix: keep the non-zero exit for branch protection, but add `if: always()` to the upload-artifact step; then classify artifact-backed blocking reviews as `changes_requested`, reserving `error` for missing/invalid artifacts.
- **“Я вайпнув `ci_runs`, а `Error`/dashes все одно лишились” на `/ci-runs`** → root cause: ingest deduplicates already-seen GitHub workflow runs by `github_url` (`CiRepository.existingRunKeys()` + `existingKeys.has(run.html_url)`) and won’t re-persist the same Actions run unless a new run (with a new `html_url`) produces an artifact. Deleting rows can still result in the same historical Actions run being re-ingested as “status-only” if the workflow still doesn’t upload the artifact for non-zero exits. Fix: re-export/update the workflow to always upload artifact on non-zero exit, then trigger a fresh Actions run (new `html_url`) and `Refresh` ingest.
- **CI Runs page shows `Something went wrong` / `Not Found` with the GitHub docs URL for `list-workflow-runs-for-a-workflow`** → root cause: `actions.listWorkflowRuns({ workflow_id: '<filename>' })` returns **HTTP 404 when that workflow YAML is not present in the repo** (never installed, PR not merged, or deleted). That is *not* the same as "workflow exists, zero runs" (200 + `[]`). `GET /ci-runs` runs ingest first (`modules/ci/ingest.ts` → `listWorkflowRuns`), so a 404 aborted the whole list and the client painted `ErrorState` instead of the empty view. AC-28 only requires failing loudly on **403** (`actions:read`); a missing workflow is an empty source. Fix: map 404 → `[]` in `OctokitGitHubClient.listWorkflowRuns` (`adapters/github/octokit.ts:418`) before `throwIfActionsForbidden`; leave 403 as `GitHubActionsReadScopeError`.
- **A `fastify-type-provider-zod` querystring/body validation failure returns HTTP 422, not 400** — an integration test asserting `expect(res.statusCode).toBe(400)` on a bad query (`?days=abc`, `?days=9999`) fails with `expected 422 to be 400`. The rejection happens at the route boundary from the zod `querystring`/`body` schema (e.g. the eval-dashboard `DaysQuery` allow-list, `modules/eval/routes.ts:37`) before the handler runs. When asserting "rejected at the boundary (a 4xx)", assert `422` (or a `>=400 && <500` range), never `400`. This is distinct from a handler that *throws* `BadRequestError` (400) itself — schema-level validation (422) and hand-thrown domain errors (400) are different status codes in this stack.
- **`error TS1160: Unterminated template literal` pointing at the LAST line of a file with no template literals in it** → root cause: a `/** ... */` doc comment contained a literal glob-style path with a `*/` substring inside it (e.g. writing the path `modules` + `/*` + `/routes.ts` back-to-back in prose) — the `*/` closes the JSDoc comment early, so the rest of the comment text (including its closing backtick) gets scanned as real code, and the orphaned backtick is parsed as an unterminated template literal running to EOF. The error location (end of file) is nowhere near the actual typo, which makes it look like a stale incremental-build/cache issue instead — `npx tsc --noEmit <that one file>` reproduces it in isolation and rules that out in seconds. Fix: never let the two characters `*` immediately followed by `/` appear inside a `/** */` block comment's text (rephrase glob-like paths to avoid it, e.g. "module's `routes.ts` file" instead of a literal `*/`-containing glob).
- **Adding a required field to a vendored zod contract (e.g. `status` on `BlastRadius`) breaks literal object-construction fixtures OUTSIDE `src/`, not just inside it.** Grepping only `server/src` for `BlastRadius\b` before the edit found zero construction sites, but `server/test/contracts.test.ts` (a top-level `test/` fixture, sibling to `src/`, not inside it) built a literal `BlastRadius.parse({...})` without `status` and failed at `pnpm test` time — typecheck alone didn't catch it because `.parse()` fixtures are plain object literals, not statically typed as the contract. When adding a required field to any `@devdigest/shared` contract, grep the WHOLE package (`server/`, not `server/src/`) for the type name, including `test/*.test.ts`, before declaring "no construction sites found."
- **`PostgresError: column "X" of relation "Y" already exists` from `pnpm db:migrate`, even though the migration's SQL hash is already present in `__drizzle_migrations`** → root cause: `drizzle-orm/postgres-js/migrator`'s `migrate()` does not decide "already applied" purely by hash-set membership — it also compares each local migration's `_journal.json` `when` (ms timestamp) against the last-applied row's `created_at`, and treats any local entry with a `when` greater than that as pending regardless of whether an identical hash already exists in the table. Hit this after manually renumbering a colliding migration file (see the orchestration note below): the new `_journal.json` entry was given a fresh `when` (current time) instead of the original one, which made the migrator re-run already-applied `ALTER TABLE ... ADD COLUMN` statements. Fix: when hand-editing `_journal.json` to fix an index/tag after a rename, copy the **original** `when` value the migration was actually applied with (find it via `SELECT created_at FROM drizzle.__drizzle_migrations ORDER BY id` and match by row position/hash) — do not regenerate a new timestamp.
