## What Works
<!-- Approaches/solutions that worked. e.g. "X via Y in src/foo.ts:42 because …" -->
- **Verifying a new page when you can't click through it: `pnpm build` + a static `t()`-key audit.** `pnpm typecheck` does NOT catch RSC/client-boundary errors or missing-i18n-key crashes; `next build` catches the former (compiles + generates every route) but still NOT the latter — a missing `next-intl` `t("a.b")` key throws only at *render*, so it slips past both typecheck and build. The cheap guard is a node script that flattens `messages/en/<ns>.json` into dotted paths and checks every `t("…")` literal in the route's files resolves (treat `t(\`prefix.${x}\`)` template keys by their static prefix — confirm a child exists). Used to validate the whole `/skills` tree after an agent built it without runtime checks.

## What Doesn't Work
<!-- Dead ends & antipatterns. The most valuable & most-skipped section. -->
- **`AppShell` calls `useRouter()` internally (via `useGlobalShortcuts`) — rendering it in RTL tests crashes with "invariant expected app router to be mounted".** The fix: mock the shell away entirely in the test file: `vi.mock("../../../../components/app-shell", () => ({ AppShell: ({ children }) => <>{children}</> }))`. Use this pattern for any page-level component test that wraps with `AppShell`; don't try to satisfy the router requirement (it pulls in too many Next.js internals). See `app/conventions/_components/ConventionsView/ConventionsView.test.tsx`.
- **Early-return branches that replace the full page layout for edge states break the visible chrome.** When a page has fixed structural elements (header, toolbar), handling loading/empty/error via separate `return (<AppShell>…stripped layout…</AppShell>)` branches causes those elements to disappear — visually jarring and inconsistent with the design. Pattern: keep one render path with the full structure always mounted; vary only the content *inside* the card/list area for each state. Demonstrated in `app/conventions/_components/ConventionsView/ConventionsView.tsx` — the multi-branch version broke the UI; collapsed to a single return fixed it.
- **`apiFetch` (`src/lib/api.ts`) force-sets `content-type: application/json` on any non-null body, which breaks `FormData` uploads** — the browser must set `multipart/form-data; boundary=…` itself, and an explicit JSON content-type overrides it, so the server can't parse the parts. Fix: skip the JSON content-type when `init.body instanceof FormData` (do NOT set content-type at all for FormData). The skill-import upload (`api.upload` → `POST /skills/import/preview`) depends on this.
- The client has its own vendored copy of shared contracts at `client/src/vendor/shared/contracts/` that is **not auto-synced** with `server/src/vendor/shared/contracts/`. Adding a field to the server copy alone passes server typecheck but breaks client typecheck. Always mirror changes to both copies (`trace.ts` and `platform.ts` were both affected in the cost feature).
- **`AgentCard`'s `skillCount` prop existed in the component signature (`AgentCard.tsx:22`) but was never passed at either call site** (`AgentsListView.tsx`, `agents/[id]/page.tsx`) — the skill-count badge silently never rendered in the app despite a passing test that explicitly set the prop. A prop being tested in isolation doesn't guarantee it's wired at the real call sites; grep all render call sites, not just the component file, when a "why doesn't X show up" bug turns out to be a missing field.

## Codebase Patterns
<!-- Conventions & architectural decisions, with the "why". -->
- **Two distinct POST endpoints for repo "refresh" do completely different things — don't conflate them.** `POST /repos/:id/refresh` enqueues a heavy **repo clone job** (used by code indexing / repo-intel). `POST /repos/:id/poll` is a lightweight **PR-list sync from GitHub** (updates title, headSha, additions, deletions, filesCount, status). The UI Refresh button must call `/poll`; `useRefreshRepo` in `core.ts:82` was previously wired to `/refresh`, which is why pressing Refresh never updated the PR list. See `server/src/modules/polling/routes.ts` and `server/src/modules/repos/service.ts`.
- Components shared across a **route subtree** (parent + its `[param]` children) belong in the parent's `_components/` folder, not in `src/components/`. Example: `SeverityChips` is used by both `pulls/_components/PRRow` and `pulls/[number]/_components/RunHistory`, so it lives at `pulls/_components/SeverityChips/`. Only promote to `src/components/` when used across unrelated routes.
- **Lazy hover-fetch pattern** (React Query): use `enabled: hovered && !!id && hasData` to defer the network call until the user actually hovers over the trigger. Pair with `staleTime: 30_000` to avoid refetching on repeated hovers within 30 s. Implemented in `pulls/_components/PRRow/PRRow.tsx` for the findings popover — avoids N requests on initial list render.
- **Mismatched hover zones between a block wrapper and an inline-flex child cause popover to never show** — `PRRow` tracked `chipsHovered` on a full-width block `<div>` around `SeverityChips` to gate the lazy fetch. `SeverityChips` tracked its own `hovered` on the smaller `inline-flex` chips area to show the popover. The fetch fired from the large zone; the popover showed from the small zone — they rarely overlapped in practice. Fix: fetch eagerly (gated by `hasFindings`, not by hover) so findings are already loaded when the user hovers the chips. See `PRRow.tsx` — `enabled: !!pr.id && hasFindings` with no hover gate.
- **Moving hover state inside a component breaks callers that depended on external hover** — `RunSeverityChips` used `enabled: hovered` for lazy React Query fetch, where `hovered` came from its own `useState`. When the hover zone (+ popover) moved inside `SeverityChips`, `RunSeverityChips` lost the hover signal and the fetch never fired. Fix: move the data fetch one level up to `RunHistory` (one `useQuery` per PR page, keyed by `prId`) and pass findings down as props. React Query deduplication still guarantees a single network request even if multiple child components share the same `queryKey`. See `RunHistory.tsx:findingsByRunId`.
- **Absolutely-positioned popovers inside table rows are clipped by `overflow: hidden`** — `FindingsPopover` used `position: absolute` relative to `SeverityChips`'s wrapper, which is inside a PR list row. The row (or a grid ancestor) has `overflow: hidden`, so the popover was cut off. Fix: use `createPortal(card, document.body)` + `position: fixed` with coordinates from `ref.current.getBoundingClientRect()` captured on mouseenter. The portal escapes all clipping ancestors. See `FindingsPopover.tsx` + `SeverityChips.tsx:handleMouseEnter`.
- **`pointerEvents: none` on hover popovers** — when a popover is absolutely positioned over its trigger and the popover itself doesn't need to be interactive, set `pointerEvents: none`. Without it, moving the mouse from the trigger div into the popover card fires `mouseLeave` on the trigger and immediately hides the popover, causing flicker. See `FindingsPopover.tsx:44`. If the popover needs to be interactive (links, buttons), extend the hover zone to cover both trigger + popover instead.
- **Filter badge counts must come from the raw (unfiltered) array, not the filtered result** — in `FindingsPanel`, `severityCounts` is computed from `findings` (the full prop), not from `shown` (the filtered output). If it used `shown`, clicking a chip would mutate the counts on all other chips to reflect only the filtered subset, making them jump around. Computing from the source array keeps chip labels stable regardless of the active filter. See `FindingsPanel.tsx:32-39`.
- **Dynamic `Icon` component lookup requires two casts** — `Icon[name as keyof typeof Icon] as React.ComponentType<{ size?: number }>`. The first cast (`keyof typeof Icon`) satisfies the TS index signature; the second (`as React.ComponentType<...>`) is needed because the namespace type is a union of all icon shapes and TS cannot infer JSX callability from it. Without both casts you get either an index error or "JSX element type does not have any construct or call signatures". See `FindingsPanel.tsx:71`.
- **`useActiveRepo()` from `lib/repo-context.tsx` is the correct hook for any page that needs the active repo** — returns `{ repoId, activeRepo, reposLoaded }`, already synced with the top-left repo selector. Do not call `useRepos()[0]` directly (misses URL-path priority and localStorage persistence). Edge-state pattern: while `!reposLoaded` show a skeleton; when loaded and `!repoId` show an empty state ("Connect a repo first"). See `app/conventions/_components/ConventionsView/ConventionsView.tsx` for the implemented pattern.
- **Course lesson scaffolding: check `helpers.ts` and `messages/en/` before wiring a new route** — When adding a route that corresponds to a future lesson (conventions, eval-dashboard, etc.), `client/src/components/app-shell/helpers.ts:activeKeyFor` likely already returns the correct sidebar key for that path, and a skeleton i18n file may already exist under `client/messages/en/`. Found both pre-wired for `/conventions` before adding any code. Creating duplicates or overwriting them causes subtle drift, so always read those two locations first.
- **`_components/` promotion to `src/components/` is skipped when the refactor would exceed the task scope** — the rule "promote to `src/components/` when used across unrelated routes" yields to CLAUDE.md's "no refactor beyond what the task requires." Result: `CreateSkillModal` (under `skills/_components/`) is cross-imported by `ConventionsView` without being moved. The cross-import is intentional, not a bug. If a future task widens the sharing further, that is the moment to promote. See `app/conventions/_components/ConventionsView/ConventionsView.tsx:11`.

- **Convention title edits are client-side-only state, mirroring accept/reject — there is no `PATCH /repos/:repoId/conventions/:id` endpoint.** `ConventionsView.tsx` keeps an `edits: Record<string, string>` map (id → overridden title) alongside `statuses`; `toConvention()` applies the override for display, and `acceptedCandidates` applies it again before `buildSkillBody()` so the edited text reaches the generated skill. If a future lesson needs edits to survive a rescan/refresh, add the PATCH route in `server/src/modules/conventions/routes.ts` first — right now a `rescan` overwrites the query cache and silently drops any in-progress edits.

- **The PR detail page's scroll container is `<main>` (`overflow-y: auto`), not `window`** — `window.scrollY` stays `0` even when the page visibly scrolls, because `<main>` (inside `AppShell`) is the actual scrollable element (`clientHeight` ≪ `scrollHeight`; `<body>`/`<html>` are non-scrolling). Any manual verification of scroll behavior (browser devtools, tests) must check `document.querySelector('main').scrollTop`, not `window.scrollY`. `Element.scrollIntoView()` itself works fine either way — it walks up to the nearest scrollable ancestor automatically — but *checking* whether it worked requires reading the right property. Found while verifying Smart Diff's finding-click-to-scroll feature.
- **`router.replace(url)` in the Next.js App Router scrolls to top by default — it fights a manual `scrollIntoView()` triggered by the same navigation.** Built a "click a badge on one tab → switch tab + scroll to a specific card" flow (Smart Diff → Findings tab, `_components/DiffTab`/`FindingsPanel`); the tab-switch `router.replace()` and the target card's `element.scrollIntoView()` effect both fired on the same navigation, and Next's own scroll-restoration always won, landing on the first card instead of the target. Fix: pass `router.replace(url, { scroll: false })` for that specific navigation so the app's own scroll effect is the only one acting. Don't disable `scroll` globally — plain tab switches (Overview → Files changed) still want the default top-scroll.
- **Running `pnpm build` while `pnpm dev` is live on the same package corrupts the shared `.next/` directory** — both commands write into `.next/` (dev writes incremental chunks, build does a full optimized rebuild), and running them concurrently leaves the dev server serving stale/missing vendor-chunk references (`Cannot find module './vendor-chunks/...'` runtime error on every route). Symptom is a full-page Next.js Runtime Error dialog, not a compile error. Fix: kill the dev process, `rm -rf .next`, restart `pnpm dev`. Avoid running `pnpm build` as a "just checking" step while a dev server for the same package is running — use it only when the dev server is stopped, or accept the restart cost.

- **`diff-viewer`'s `parsePatch`/styles (`s.hunk`, `s.lineNo`, `s.lineText`, `lineRowFor`, `lineSignFor`, `chevronFor`, `AUTO_EXPAND_MAX_LINES`) are all exported and safely reusable outside the diff-viewer folder — `CodeLine` itself is not** (it's tightly coupled to `DiffCommentApi`/comment threads via `commentTargetFor`/`InlineComposer`). When a new feature needs to render raw diff lines without commenting (e.g. Smart Diff's per-role file rows), reuse the exported helpers/styles directly and build a small local line renderer — don't try to reuse `CodeLine` itself or pass it a fake/empty `commenting` prop. See `SmartDiffViewer/SmartDiffViewer.tsx`.
- **A route contract that omits a field (here `SmartDiffFile` has no `patch`) doesn't mean the UI can't show that content — check if the data is already loaded elsewhere on the page first.** Built Smart Diff's file rows initially as a static summary (path + stat + badges) because the `/smart-diff` route's `SmartDiffFile` has no `patch`. That was a mistake: `usePullDetail` already fetches the full `PrFile[]` (with `patch`) for the same PR, and it's already passed into `DiffTab`. Fix: thread `files: PrFile[]` into `SmartDiffViewer` and cross-reference by `path` — no backend/contract change needed. Before declaring a UI capability "not possible" because one API response lacks a field, check sibling data already on the page.
- **A written implementation plan is not "done" until checked against its own explicit spec language, not just "does it toggle/render."** The plan said file rows "mirror `FileCard`'s collapse pattern" (i.e. expand to show the diff), but the first pass only built a collapsed summary row with no way to see the diff at all — a real functional gap, not a polish issue. It passed typecheck, build, unit tests, and even a live browser click-through (toggle + badge navigation) because none of those checks compare rendered output against the actual target (a mockup screenshot, in this case). When a plan references a specific existing component's *behavior* (not just its visual style), verify the new component actually replicates that behavior, not just its collapsed-state appearance.

- **A single "find the finding for this line" helper can't correctly serve both range-highlighting and single-badge-placement — they need different match semantics from the same call site.** Smart Diff's per-line severity stripe needed to span a finding's whole `start_line..end_line` range, but the clickable badge must render exactly once (at the top of the range), not once per matched line. Initially matched `line === start_line` only (so both stripe and badge were correct only on line 1 of the range, and the stripe never covered the rest). Fix: `findingForLine` (`SmartDiffViewer/helpers.ts`) matches the **inclusive range** (`start_line <= line <= end_line`) for stripe color; the caller (`SmartDiffViewer.tsx`'s `DiffLine`) additionally checks `lineNo === finding.start_line` before rendering the badge. One helper, one range-based match, two different uses of the result — don't try to encode "and also only on the first line" into the matching function itself.

- **A server-side session-window filter (Decision 2's 60s rule) doesn't automatically apply to every client-side consumer of the same findings — each place that fans a review list out into a flat findings list needs its own filter, because `FindingRecord` has no `created_at` of its own.** `assembleSmartDiff` correctly filters `reviews` to the 60s session window server-side before building `finding_lines` (used for the collapsed-row indicator dot in `SmartDiffViewer`). But `page.tsx`'s `allFindings = runs.flatMap(r => r.findings)` (used for the Findings tab / lethal-trifecta banner, intentionally *not* session-windowed — it should show every finding ever recorded) was also being passed to `SmartDiffViewer` for the per-line stripe/badge rendering, silently including stale out-of-session findings there too — a real behavioral mismatch caught only by re-reading the plan's Decision 2 against the actual data flow, not by any test or visual check. Fix: `sessionWindowFindings(reviews)` (`SmartDiffViewer/helpers.ts`) mirrors the server logic (filter reviews by 60s window, then flatten) and feeds a *separate* `smartDiffFindings` value distinct from `allFindings`. When a decision says "X window applies to feature Y," check every place Y's data is derived, not just the one where the rule was first implemented.

## Tool & Library Notes
<!-- Dependency quirks, version gotchas, env/config oddities. -->
- `NEXT_PUBLIC_*` env vars are inlined into the bundle at **start time** (`next dev` / `next build`), not at request time. Changing a `NEXT_PUBLIC_*` value in `.env` requires a dev-server restart to take effect — a page refresh is not enough.

## Recurring Errors & Fixes
<!-- Error signature → root cause → fix. -->
- **`textDecoration: underline` has no effect on SVG icons inside `display: inline-flex`** — root cause: `text-decoration` only draws under text nodes, not SVG children. Fix: use `borderBottom: "1px solid/dashed/dotted <color>"` + `paddingBottom: 2` on the `inline-flex` span instead — border applies to the element box and covers both icon and number. See `SeverityChips.tsx` interactive chip style.
- **`TS2322: Type 'number | null | undefined' is not assignable to type 'number | null'` in `RunHistory.test.tsx`** — root cause: `RunSummary` gained a new `nullable()` field (not `nullish()`), so the `Partial<RunSummary>` factory in the test fixture no longer satisfies the full type. Fix: add the new field with an explicit `null` default inside the `run()` factory function (`RunHistory.test.tsx:30`).
