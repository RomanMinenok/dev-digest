## What Works
<!-- Approaches/solutions that worked. e.g. "X via Y in src/foo.ts:42 because …" -->
- **Verifying a new page when you can't click through it: `pnpm build` + a static `t()`-key audit.** `pnpm typecheck` does NOT catch RSC/client-boundary errors or missing-i18n-key crashes; `next build` catches the former (compiles + generates every route) but still NOT the latter — a missing `next-intl` `t("a.b")` key throws only at *render*, so it slips past both typecheck and build. The cheap guard is a node script that flattens `messages/en/<ns>.json` into dotted paths and checks every `t("…")` literal in the route's files resolves (treat `t(\`prefix.${x}\`)` template keys by their static prefix — confirm a child exists). Used to validate the whole `/skills` tree after an agent built it without runtime checks.
- **Surfacing agent-version skew on the Evals tab without relaxing the AC-31 metric gate.** When `current.traces_total === 0` but prior-version `latest_run`s exist, keep metric cards as `—` and instead: (1) `fullyStale` banner naming both versions (`EvalsTab.tsx` + `EvalsTabView` `staleBanner`), (2) mute per-case pass/fail icons and show `last measured on v{M}` via `isRunStaleForAgent` (`helpers.ts`), (3) replace `N / M passing` with `not scored on v{N}` and count passes only for `!isRunStaleForAgent` runs. AC-33's cross-version `latest_run` stays the history source — the bug was *rendering* it as a current-config verdict. Do not fill metrics from last-measured aggregates; that would silently attribute an older agent score to the new config.
- **(2026-07-15 correction / supersedes the "Do not fill metrics…" clause above under SPEC-04)** After `EvalDashboard.current` became last-measured (not live version), blanking cards to `—` fought the wire data: the numbers are already there and useful. Product choice: keep showing `dashboard.current` aggregates, gate `fullyStale` on `measured_version !== agent.version` (`EvalsTab.tsx`), and label them explicitly (`staleVersionBanner` / `metricsSubtitleUnmeasured` in `eval.json`) while cases stay `not scored on v{N}` + muted. **Attribution is the gate, not emptiness** — silent old scores are wrong; labelled last-measured scores are fine.
- **Live per-case progress + header metrics on "Run all" without SSE: sequence single-case `mutateAsync`s, don't use the batch POST.** `POST /agents/:id/eval-runs` with `{}` runs cases sequentially server-side but returns only after the whole pass — the client then has no mid-batch signal, so cards and metric cards stay frozen until the end. `EvalsTab.tsx`'s `handleRunAll` instead loops `mutateAsync({ case_ids: [id] })`: each success already invalidates both `eval-cases` and `eval-dashboard` (`hooks/evals.ts` `useInvalidateEvalQueries`), so pass/fail icons and Recall/Precision/Citation refresh after every case; `runningCaseId` from mutation `variables` lights the in-flight card. `runAllActive` local state must wrap the whole loop — `isPending` only covers one HTTP call, so it drops briefly between cases and cannot alone drive the bulk "Running…" button. Mid-batch metrics are a partial aggregate over completed current-version runs only (server `dashboard` / `aggregateRunRows`); that is expected, not a bug. Prefer this over polling-the-batch-POST or inventing SSE unless the N round-trips become a real problem.

## What Doesn't Work
<!-- Dead ends & antipatterns. The most valuable & most-skipped section. -->
- **Putting the whole `useMutation()` return object in `useCallback` / `useEffect` deps causes an infinite mutate → re-render → mutate loop.** `ExportWizard.tsx` wrapped `previewMutation.mutate(...)` in `fetchPreview` with `[…, previewMutation]` and an effect keyed on `fetchPreview` that fires whenever `state.step === 1`. TanStack Query v5's `mutate` function is stable, but the **mutation result object** is a new reference every render (pending/error/data updates). Each `mutate` updates that object → `fetchPreview` identity changes → effect re-fires → another POST → Fastify rate-limit toast storm while Preview stays on "Generating…". Fix (planned, not yet applied): destructure `const { mutate: previewMutate, … } = useCiPreview()` and depend only on the stable `mutate`, plus a ref of the last-fetched `previewInput` key so the same body is not re-POSTed. Do not list `previewMutation` (the hook return) in effect deps.
- **`EvalRunDetailModal` + case-card copy still treat a provider-failure run as an empty finding list.** When `EvalService.runCases` catches a model throw, it stores the error message as a *string* `actual_output` (not `Finding[]`) with null recall/precision. The UI path collapses that: `countActualFindings` (`EvalsTab/helpers.ts`) returns `0` for non-arrays → card subtitle `expected N finding, got 0`; `parseActualFindings` likewise returns `[]` → modal "No findings produced" / ACTUAL (0). Confirmed 2026-07-15 after a Gemini model switch: every case card and View-result looked like a scoring miss while the wire truth was `actual_output: "400 Provider returned error"`. Rule: when diagnosing "evals fail after a model change," first check whether `latest_run.actual_output` is a **string** (provider/run error) vs an **array** (scored output); the modal does not surface the former today.
- **`api.get<SomeContract[]>` / `api.post<SomeContract>` do not validate the response against the Zod contract — a server that returns Drizzle camelCase rows typechecks against a snake_case `EvalCase` generic and still compiles.** SPEC-03's Evals tab looked green after T14–T21 (`pnpm typecheck` PASS) while `c.input_diff` / list row fields were actually `undefined` at runtime. Client-side you cannot detect this without a live response or a mapper test; treat a new endpoint's first browser/curl of the real JSON shape as part of "done," not optional polish. Server-side fix was `toEvalCaseDto` (`server/src/modules/eval/service.ts`); the lasting client lesson is: the generic on `api.*` is documentation, not a boundary check.
- **Rendering an unmeasured metric as `0%` — "no data" and "scored zero" are different facts and must not share a glyph.** SPEC-03's Evals tab gated its "unavailable" (`—`) state on `dashboard.cases_total === 0`, i.e. *no cases exist*. But an agent can have cases and have never run them: the server then returns real metrics of `0` alongside `traces_total: 0`, and the cards read `0% / 0% / 0%` — an agent that has never been measured presents as a catastrophically failing one. The correct gate is the count of *runs behind the aggregate* (`dashboard.current.traces_total === 0`), not the count of cases (`EvalsTabView.tsx`). **Rule: a metric card's empty state must key off the count of observations, never off the count of things that could theoretically be observed** — and when a server returns `0` for an unpopulated aggregate, treat the accompanying `*_total` as the real signal, because `0` is a legitimate value the metric can also take.
- **A later, unrelated feature reintroduced an already-fixed scroll race by adding a THIRD actor that scrolls the same container.** `434c239` fixed Smart Diff's badge→finding-card centering by making `FindingsPanel` the sole scroll owner (deferred rAF + `ReviewRunAccordion` skipping its own `scrollIntoView` when `targetFindingId` is set). Commit `349aadd` later added a tab-scroll-persistence effect in `page.tsx` (`React.useEffect(() => { main.scrollTop = tabScrollTop.current[tab] ?? 0; ... }, [tab])`) that runs on every tab switch, including a badge-click switch to the findings tab — it doesn't know about `findingId` and jumps `main.scrollTop` to a stale saved offset from the *previous* findings-tab visit before `FindingsPanel`'s smooth-centering scroll takes over. Confirmed via a `requestAnimationFrame` scrollTop trace: the buggy version jumps instantly to the stale offset then crawls back over ~1.1s to the correct centered position — same final resting `scrollTop` as the fixed version, so a single before/after position check misses it entirely (see Tool & Library Notes entry below). Fix: `page.tsx`'s restore effect skips the jump when `tab === "findings" && findingId` (a pending target) is true, leaving `FindingsPanel` as sole owner — same principle as the original fix, just applied to a new actor. **General rule: when adding any effect that sets `scrollTop`/`scrollIntoView` on a shared scroll container, grep for existing scroll owners on that same container before assuming there's only one.**
- **`AppShell` calls `useRouter()` internally (via `useGlobalShortcuts`) — rendering it in RTL tests crashes with "invariant expected app router to be mounted".** The fix: mock the shell away entirely in the test file: `vi.mock("../../../../components/app-shell", () => ({ AppShell: ({ children }) => <>{children}</> }))`. Use this pattern for any page-level component test that wraps with `AppShell`; don't try to satisfy the router requirement (it pulls in too many Next.js internals). See `app/conventions/_components/ConventionsView/ConventionsView.test.tsx`.
- **Early-return branches that replace the full page layout for edge states break the visible chrome.** When a page has fixed structural elements (header, toolbar), handling loading/empty/error via separate `return (<AppShell>…stripped layout…</AppShell>)` branches causes those elements to disappear — visually jarring and inconsistent with the design. Pattern: keep one render path with the full structure always mounted; vary only the content *inside* the card/list area for each state. Demonstrated in `app/conventions/_components/ConventionsView/ConventionsView.tsx` — the multi-branch version broke the UI; collapsed to a single return fixed it.
- **`apiFetch` (`src/lib/api.ts`) force-sets `content-type: application/json` on any non-null body, which breaks `FormData` uploads** — the browser must set `multipart/form-data; boundary=…` itself, and an explicit JSON content-type overrides it, so the server can't parse the parts. Fix: skip the JSON content-type when `init.body instanceof FormData` (do NOT set content-type at all for FormData). The skill-import upload (`api.upload` → `POST /skills/import/preview`) depends on this.
- The client has its own vendored copy of shared contracts at `client/src/vendor/shared/contracts/` that is **not auto-synced** with `server/src/vendor/shared/contracts/`. Adding a field to the server copy alone passes server typecheck but breaks client typecheck. Always mirror changes to both copies (`trace.ts` and `platform.ts` were both affected in the cost feature).
- **`AgentCard`'s `skillCount` prop existed in the component signature (`AgentCard.tsx:22`) but was never passed at either call site** (`AgentsListView.tsx`, `agents/[id]/page.tsx`) — the skill-count badge silently never rendered in the app despite a passing test that explicitly set the prop. A prop being tested in isolation doesn't guarantee it's wired at the real call sites; grep all render call sites, not just the component file, when a "why doesn't X show up" bug turns out to be a missing field.
- **One `useMutation` instance shared across N buttons broadcasts its `isPending` to all N, making a single action look like it triggered all of them.** `EvalsTab.tsx` called `useRunEvals(agent.id)` once and passed the same `runEvals.isPending` as `runningAll` into every per-case Play button in `EvalsTabView.tsx` (`disabled={runningAll} loading={runningAll}` on each card, plus the "Run all" button) — the server correctly ran only the requested `case_id` (`service.runCases` filters by `caseIds` when provided), but clicking one card's Run button made every card's button spin, reading as "running all evals" even though only one was. User-reported symptom: "запустити один евал по його картці, то запускаються всі". Fix: derive per-target pending state from the shared mutation's own `variables` (react-query keeps the in-flight call's `variables` on the mutation object while `isPending`) — `runningCaseId = isPending && variables?.case_ids?.length === 1 ? variables.case_ids[0] : null`, `isRunningAll = isPending && variables === undefined` — and key each button's `loading` off its own identity, not the shared flag. General rule: when several UI triggers share one mutation/query hook instance, never wire a bare `isPending`/`isFetching` straight to every trigger's loading prop — derive an identity-specific signal from `variables`/`queryKey` first, or the loading UI will lie about scope even when the underlying call was scoped correctly.

- **Two nested components each firing their own `scrollIntoView()` on the same target change race and land on a random final position.** `ReviewRunAccordion` scrolled its whole container (`block: "start"`) the instant it flipped `open` (before the newly-mounted child had laid out), while its child `FindingsPanel` separately scrolled the target finding card (`block: "start"`) right after mounting — both effects fired off the same `targetFindingId`/`targetRunId` change, and the accordion's expand animation shifted layout mid-scroll. Symptom: clicking a Smart Diff badge sometimes landed the card off-screen. Fix: only one component should own the scroll for a given navigation — `ReviewRunAccordion` now skips its own `scrollIntoView` whenever a `targetFindingId` is present (`ReviewRunAccordion.tsx`), leaving `FindingsPanel` as the sole scroller, deferred one `requestAnimationFrame` so it scrolls after the accordion's expand has committed, and using `block: "center"` (not `"start"`) so short elements don't hug the viewport edge. When two ancestors both want to scroll to "roughly the same place" on the same state change, pick exactly one owner.
- **Empty Evals metric cards (`—`) after a page refresh while case rows still show pass/fail is usually agent-version skew, not a broken fetch or a wrong unavailable gate.** `GET /agents/:id/eval-dashboard`'s `current` aggregates only runs at `agents.version` (AC-31; `EvalService.dashboard` → `runsForAgentVersion`); case rows pull `latest_run` across *every* version (`list` → `latestRunPerCase`). Any config save bumps the version (`agents/repository.ts`, everything except `enabled`), so a measured agent that has not been re-eval'd on the new version returns `current.traces_total: 0` → `EvalsTabView`'s gate correctly paints `—`, while the list still looks scored. Confirmed live on agent `0992acbe-…`: agent `version: 3`, all three `recent_runs` / `latest_run`s at `agent_version: 2`. **Do not "fix" the empty header by relaxing the `traces_total === 0` gate** (that gate is right — see the unmeasured-vs-0% entry above); first curl-compare `agent.version` to `latest_run.agent_version` / `recent_runs[].agent_version`. Immediate unblock: re-run evals on the current version. UX follow-ups (clearer empty copy / last-measured fallback) still undecided — see Open Questions.
- **(2026-07-15 correction of the entry above + partial obsolescence of the "Surfacing agent-version skew…`traces_total === 0`" What Works note)** SPEC-04 redefined `EvalDashboard.current` as the **latest measured** version aggregate (`measured_version` / `groupVersionRuns` → `measuredVersions` in `server/src/modules/eval/service.ts`), deliberately *not* live `agents.version` — so after a config bump with only prior-version runs, `current.traces_total` stays `> 0` and the cards paint last-measured 100%s. The Agent Editor Evals tab still derives `fullyStale` from `traces_total === 0` (`EvalsTab.tsx`), so the stale banner + `not scored on v{N}` + metric `—` path silently never fires, while per-case `isRunStaleForAgent` still shows `last measured on v{M}` and `passingCount` correctly stays `0`. Symptom matched live: 100% Recall/Precision/Citation, no banner, `0 / 9 passing`, all rows stale. **Rule: after a shared DTO field's meaning changes for one consumer, grep every other consumer of that field** — `AgentEvalView` already gates skew on `measured_version !== agent.version`; the Evals tab must do the same (and still force metric unavailable when skewing) rather than trusting `traces_total === 0` as "unmeasured on live config." Fix proposed this session, not yet applied.
- **(2026-07-15 follow-up — fixed)** Applied the `measured_version !== agent.version` gate in `EvalsTab.tsx`; user chose **labelled last-measured metrics** (keep `dashboard.current` values + banner/subtitle), not blank `—`. The "force metric unavailable" clause in the correction above is obsolete — see the What Works addendum.

## Codebase Patterns
<!-- Conventions & architectural decisions, with the "why". -->
- **Pure helpers needed by both the PR route (`FindingsPanel`) and the Agent Editor (`EvalCaseModal`) belong in `src/lib/`, not in either route's `_components/`.** SPEC-03 originally left `slugifyTitle` / `sliceDiffToFile` / `expectedFromFinding` under `agents/[id]/…/EvalCaseModal/helpers.ts` and had FindingsPanel deep-import them — architecture-reviewer flagged it High against `client/CLAUDE.md`'s route-private `_components` rule. Fix: `client/src/lib/eval-capture.ts` (next to `eval-prefill.ts`); leave modal-only JSON helpers route-local. Distinct from the older "CreateSkillModal cross-import kept intentionally" exception (`ConventionsView`) — that was a whole modal UI; shared *pure capture transforms* should promote to `lib/` immediately.
- **A written Development Plan's specific icon names and file paths are not guaranteed to exist — verify against the actual repo before implementing, don't just trust the prose.** The Blast Radius plan (`docs/plan/blast_radius_plan.md`) specified icon `"GitFork"` for the new tab and file path `client/src/messages/en/prReview.json`; neither was correct — `client/src/vendor/ui/icons.tsx` has no `GitFork` in its lucide-react import list (had to substitute `"Workflow"`, an icon that IS registered), and the real i18n file lives at `client/messages/en/prReview.json` (no `src/`). Before wiring a plan-specified icon name into a `Tabs`/`Icon.X` usage, grep `client/src/vendor/ui/icons.tsx`'s export list first; before writing to a plan-specified file path, `find`/`ls` it first.
- **Two distinct POST endpoints for repo "refresh" do completely different things — don't conflate them.** `POST /repos/:id/refresh` enqueues a heavy **repo clone job** (used by code indexing / repo-intel). `POST /repos/:id/poll` is a lightweight **PR-list sync from GitHub** (updates title, headSha, additions, deletions, filesCount, status). The UI Refresh button must call `/poll`; `useRefreshRepo` in `core.ts:82` was previously wired to `/refresh`, which is why pressing Refresh never updated the PR list. See `server/src/modules/polling/routes.ts` and `server/src/modules/repos/service.ts`.
- Components shared across a **route subtree** (parent + its `[param]` children) belong in the parent's `_components/` folder, not in `src/components/`. Example: `SeverityChips` is used by both `pulls/_components/PRRow` and `pulls/[number]/_components/RunHistory`, so it lives at `pulls/_components/SeverityChips/`. Only promote to `src/components/` when used across unrelated routes.
- **Lazy hover-fetch pattern** (React Query): use `enabled: hovered && !!id && hasData` to defer the network call until the user actually hovers over the trigger. Pair with `staleTime: 30_000` to avoid refetching on repeated hovers within 30 s. Implemented in `pulls/_components/PRRow/PRRow.tsx` for the findings popover — avoids N requests on initial list render.
- **Mismatched hover zones between a block wrapper and an inline-flex child cause popover to never show** — `PRRow` tracked `chipsHovered` on a full-width block `<div>` around `SeverityChips` to gate the lazy fetch. `SeverityChips` tracked its own `hovered` on the smaller `inline-flex` chips area to show the popover. The fetch fired from the large zone; the popover showed from the small zone — they rarely overlapped in practice. Fix: fetch eagerly (gated by `hasFindings`, not by hover) so findings are already loaded when the user hovers the chips. See `PRRow.tsx` — `enabled: !!pr.id && hasFindings` with no hover gate.
- **Moving hover state inside a component breaks callers that depended on external hover** — `RunSeverityChips` used `enabled: hovered` for lazy React Query fetch, where `hovered` came from its own `useState`. When the hover zone (+ popover) moved inside `SeverityChips`, `RunSeverityChips` lost the hover signal and the fetch never fired. Fix: move the data fetch one level up to `RunHistory` (one `useQuery` per PR page, keyed by `prId`) and pass findings down as props. React Query deduplication still guarantees a single network request even if multiple child components share the same `queryKey`. See `RunHistory.tsx:findingsByRunId`.
- **Absolutely-positioned popovers inside table rows are clipped by `overflow: hidden`** — `FindingsPopover` used `position: absolute` relative to `SeverityChips`'s wrapper, which is inside a PR list row. The row (or a grid ancestor) has `overflow: hidden`, so the popover was cut off. Fix: use `createPortal(card, document.body)` + `position: fixed` with coordinates from `ref.current.getBoundingClientRect()` captured on mouseenter. The portal escapes all clipping ancestors. See `FindingsPopover.tsx` + `SeverityChips.tsx:handleMouseEnter`.
- **`pointerEvents: none` on hover popovers** — when a popover is absolutely positioned over its trigger and the popover itself doesn't need to be interactive, set `pointerEvents: none`. Without it, moving the mouse from the trigger div into the popover card fires `mouseLeave` on the trigger and immediately hides the popover, causing flicker. See `FindingsPopover.tsx:44`. If the popover needs to be interactive (links, buttons), extend the hover zone to cover both trigger + popover instead.
- **Filter badge counts must come from the raw (unfiltered) array, not the filtered result** — in `FindingsPanel`, `severityCounts` is computed from `findings` (the full prop), not from `shown` (the filtered output). If it used `shown`, clicking a chip would mutate the counts on all other chips to reflect only the filtered subset, making them jump around. Computing from the source array keeps chip labels stable regardless of the active filter. See `FindingsPanel.tsx:32-39`.
- **Dynamic `Icon` component lookup requires two casts** — `Icon[name as keyof typeof Icon] as React.ComponentType<{ size?: number }>`. The first cast (`keyof typeof Icon`) satisfies the TS index signature; the second (`as React.ComponentType<...>`) is needed because the namespace type is a union of all icon shapes and TS cannot infer JSX callability from it. Without both casts you get either an index error or "JSX element type does not have any construct or call signatures". See `FindingsPanel.tsx:71`.
- **`useActiveRepo()` from `lib/repo-context.tsx` is the correct hook for any page that needs the active repo** — returns `{ repoId, activeRepo, reposLoaded }`, already synced with the top-left repo selector. Do not call `useRepos()[0]` directly (misses URL-path priority and localStorage persistence). Edge-state pattern: while `!reposLoaded` show a skeleton; when loaded and `!repoId` show an empty state ("Connect a repo first"). See `app/conventions/_components/ConventionsView/ConventionsView.tsx` for the implemented pattern.
- **Course lesson scaffolding: check `helpers.ts` and `messages/en/` before wiring a new route** — When adding a route that corresponds to a future lesson (conventions, eval-dashboard, etc.), `client/src/components/app-shell/helpers.ts:activeKeyFor` likely already returns the correct sidebar key for that path, and a skeleton i18n file may already exist under `client/messages/en/`. Found both pre-wired for `/conventions` before adding any code. Creating duplicates or overwriting them causes subtle drift, so always read those two locations first.
- **`_components/` promotion to `src/components/` is skipped when the refactor would exceed the task scope** — the rule "promote to `src/components/` when used across unrelated routes" yields to CLAUDE.md's "no refactor beyond what the task requires." Result: `CreateSkillModal` (under `skills/_components/`) is cross-imported by `ConventionsView` without being moved. The cross-import is intentional, not a bug. If a future task widens the sharing further, that is the moment to promote. See `app/conventions/_components/ConventionsView/ConventionsView.tsx:11`.
- **Promoting a route-local `_components/` tree to `src/components/` is not done until the old directory is deleted — updating imports alone leaves an orphan twin.** SPEC-05 T25 moved `RunTraceDrawer` to `client/src/components/run-trace-drawer/` and rewired PR page + CI Runs callers, but the original 21-file tree under `repos/.../pulls/.../_components/RunTraceDrawer/` stayed on disk. plan-verifier marked T25 Partial; architecture-reviewer flagged Medium risk of future divergence. Typecheck/build stayed green because nothing imported the orphan. Rule: after a promote, `rm -rf` the old path and `rg` for the old relative import — "all callers updated" is necessary but not sufficient.

- **Convention title edits are client-side-only state, mirroring accept/reject — there is no `PATCH /repos/:repoId/conventions/:id` endpoint.** `ConventionsView.tsx` keeps an `edits: Record<string, string>` map (id → overridden title) alongside `statuses`; `toConvention()` applies the override for display, and `acceptedCandidates` applies it again before `buildSkillBody()` so the edited text reaches the generated skill. If a future lesson needs edits to survive a rescan/refresh, add the PATCH route in `server/src/modules/conventions/routes.ts` first — right now a `rescan` overwrites the query cache and silently drops any in-progress edits.

- **The PR detail page's scroll container is `<main>` (`overflow-y: auto`), not `window`** — `window.scrollY` stays `0` even when the page visibly scrolls, because `<main>` (inside `AppShell`) is the actual scrollable element (`clientHeight` ≪ `scrollHeight`; `<body>`/`<html>` are non-scrolling). Any manual verification of scroll behavior (browser devtools, tests) must check `document.querySelector('main').scrollTop`, not `window.scrollY`. `Element.scrollIntoView()` itself works fine either way — it walks up to the nearest scrollable ancestor automatically — but *checking* whether it worked requires reading the right property. Found while verifying Smart Diff's finding-click-to-scroll feature.
- **`router.replace(url)` in the Next.js App Router scrolls to top by default — it fights a manual `scrollIntoView()` triggered by the same navigation.** Built a "click a badge on one tab → switch tab + scroll to a specific card" flow (Smart Diff → Findings tab, `_components/DiffTab`/`FindingsPanel`); the tab-switch `router.replace()` and the target card's `element.scrollIntoView()` effect both fired on the same navigation, and Next's own scroll-restoration always won, landing on the first card instead of the target. Fix: pass `router.replace(url, { scroll: false })` for that specific navigation so the app's own scroll effect is the only one acting. Don't disable `scroll` globally — plain tab switches (Overview → Files changed) still want the default top-scroll.
- **Running `pnpm build` while `pnpm dev` is live on the same package corrupts the shared `.next/` directory** — both commands write into `.next/` (dev writes incremental chunks, build does a full optimized rebuild), and running them concurrently leaves the dev server serving stale/missing vendor-chunk references (`Cannot find module './vendor-chunks/...'` runtime error on every route). Symptom is a full-page Next.js Runtime Error dialog, not a compile error. Fix: kill the dev process, `rm -rf .next`, restart `pnpm dev`. Avoid running `pnpm build` as a "just checking" step while a dev server for the same package is running — use it only when the dev server is stopped, or accept the restart cost.

- **`diff-viewer`'s `parsePatch`/styles (`s.hunk`, `s.lineNo`, `s.lineText`, `lineRowFor`, `lineSignFor`, `chevronFor`, `AUTO_EXPAND_MAX_LINES`) are all exported and safely reusable outside the diff-viewer folder — `CodeLine` itself is not** (it's tightly coupled to `DiffCommentApi`/comment threads via `commentTargetFor`/`InlineComposer`). When a new feature needs to render raw diff lines without commenting (e.g. Smart Diff's per-role file rows), reuse the exported helpers/styles directly and build a small local line renderer — don't try to reuse `CodeLine` itself or pass it a fake/empty `commenting` prop. See `SmartDiffViewer/SmartDiffViewer.tsx`.
- **A route contract that omits a field (here `SmartDiffFile` has no `patch`) doesn't mean the UI can't show that content — check if the data is already loaded elsewhere on the page first.** Built Smart Diff's file rows initially as a static summary (path + stat + badges) because the `/smart-diff` route's `SmartDiffFile` has no `patch`. That was a mistake: `usePullDetail` already fetches the full `PrFile[]` (with `patch`) for the same PR, and it's already passed into `DiffTab`. Fix: thread `files: PrFile[]` into `SmartDiffViewer` and cross-reference by `path` — no backend/contract change needed. Before declaring a UI capability "not possible" because one API response lacks a field, check sibling data already on the page.
- **A written implementation plan is not "done" until checked against its own explicit spec language, not just "does it toggle/render."** The plan said file rows "mirror `FileCard`'s collapse pattern" (i.e. expand to show the diff), but the first pass only built a collapsed summary row with no way to see the diff at all — a real functional gap, not a polish issue. It passed typecheck, build, unit tests, and even a live browser click-through (toggle + badge navigation) because none of those checks compare rendered output against the actual target (a mockup screenshot, in this case). When a plan references a specific existing component's *behavior* (not just its visual style), verify the new component actually replicates that behavior, not just its collapsed-state appearance.

- **A single "find the finding for this line" helper can't correctly serve both range-highlighting and single-badge-placement — they need different match semantics from the same call site.** Smart Diff's per-line severity stripe needed to span a finding's whole `start_line..end_line` range, but the clickable badge must render exactly once (at the top of the range), not once per matched line. Initially matched `line === start_line` only (so both stripe and badge were correct only on line 1 of the range, and the stripe never covered the rest). Fix: `findingForLine` (`SmartDiffViewer/helpers.ts`) matches the **inclusive range** (`start_line <= line <= end_line`) for stripe color; the caller (`SmartDiffViewer.tsx`'s `DiffLine`) additionally checks `lineNo === finding.start_line` before rendering the badge. One helper, one range-based match, two different uses of the result — don't try to encode "and also only on the first line" into the matching function itself.

- **A server-side session-window filter (Decision 2's 60s rule) doesn't automatically apply to every client-side consumer of the same findings — each place that fans a review list out into a flat findings list needs its own filter, because `FindingRecord` has no `created_at` of its own.** `assembleSmartDiff` correctly filters `reviews` to the 60s session window server-side before building `finding_lines` (used for the collapsed-row indicator dot in `SmartDiffViewer`). But `page.tsx`'s `allFindings = runs.flatMap(r => r.findings)` (used for the Findings tab / lethal-trifecta banner, intentionally *not* session-windowed — it should show every finding ever recorded) was also being passed to `SmartDiffViewer` for the per-line stripe/badge rendering, silently including stale out-of-session findings there too — a real behavioral mismatch caught only by re-reading the plan's Decision 2 against the actual data flow, not by any test or visual check. Fix: `sessionWindowFindings(reviews)` (`SmartDiffViewer/helpers.ts`) mirrors the server logic (filter reviews by 60s window, then flatten) and feeds a *separate* `smartDiffFindings` value distinct from `allFindings`. When a decision says "X window applies to feature Y," check every place Y's data is derived, not just the one where the rule was first implemented.

- **A tab's scroll position must be persisted in the parent, not the tab's own component, because the tab unmounts on switch.** `page.tsx` renders `tab === "diff" && <DiffTab/>` (conditional mount, not CSS-hidden), so any `useState` scroll tracking inside `DiffTab`/`SmartDiffViewer` is destroyed the instant the user switches tabs. Fix: keep a `React.useRef(0)` in `page.tsx` itself (survives across tab switches since the page component never unmounts), attach a `scroll` listener to `document.querySelector('main')` (the real scroll container — see the `<main>`-not-`window` entry below) only while `tab === "diff"`, and restore `main.scrollTop` from the ref in an effect keyed on `tab` whenever it re-becomes `"diff"`. Built for the Smart-Diff-badge round trip (diff → findings → back to diff should land exactly where the user left off), see `page.tsx`'s `diffScrollTop` ref.

- **A one-shot "target and animate to it" URL param (`findingId`) must be cleared by the consumer once acted on, or every later remount replays the animation.** `page.tsx`'s `tabScrollTop` ref (extended from the diff-only `diffScrollTop` in the entry above to a `Record<tab, number>` covering both `"diff"` and `"findings"`) now restores raw scroll position on any tab revisit — but `findingId` stayed in the URL indefinitely (nothing ever cleared it), so every remount of `FindingsTab`/`ReviewRunAccordion`/`FindingsPanel` still saw a non-null `targetFindingId` and re-ran the scroll-to-card `useEffect`, re-triggering the smooth-scroll+center animation on top of the just-restored position on every single tab revisit, not just the first. Fix: `FindingsPanel` calls a new `onScrolledToTarget` callback (bubbled through `ReviewRunAccordion` → `FindingsTab` → `page.tsx`) right after firing its one-shot `scrollIntoView`, and `page.tsx` uses it to drop `findingId` from the URL (`setParams({ findingId: null }, { scroll: false })`). Only the panel that actually owns the finding (`shown.findIndex(...) >= 0`) calls it, so panels for other review runs don't prematurely clear the target before the right one has had a chance to scroll. General rule: any prop meant to trigger a one-time effect on arrival (not a fact to remember forever) needs an explicit "consumed" signal back to whoever owns that prop's source of truth — otherwise every future remount re-fires it.

## Codebase Patterns (addendum)
- **Sidebar wiring is split across TWO separate files, and only one of them makes a route actually clickable.** `components/app-shell/helpers.ts`'s `activeKeyFor` only decides which sidebar key is *highlighted* for a given path — it does NOT add an entry to the sidebar itself. The actual list of clickable sidebar items lives in `client/src/vendor/ui/nav.ts` (a `NAV` array). Adding a new route and only updating `activeKeyFor` (as the project-context implementation plan's T14 task literally scoped it) leaves the new page unreachable from the UI — the route works if you type the URL directly, but there's no link to it. When wiring a brand-new top-level route into the sidebar, update `nav.ts` as well as `activeKeyFor`, even if a task/plan only names the latter.

- **Adding a new tab to `AgentEditor` requires updating a SECOND, separate file: `app/agents/[id]/page.tsx`'s `VALID_TABS` allowlist.** `AgentEditor.tsx`'s own tab switch/`TABS` array is not the only gate — `page.tsx:15` (`const VALID_TABS = ["config", "skills"]`) validates the `?tab=` query param before it ever reaches `AgentEditor`, silently falling back to `"config"` for any tab name not in that array. Adding T15's new `"context"` tab to `AgentEditor.tsx`/`constants.ts` alone left the tab fully built but unreachable via URL/deep-link. Same *shape* of bug as the `nav.ts`/`activeKeyFor` sidebar split noted above — two independent gate files for one feature, and a plan/task that only names one of them will compile clean and look "done" while being unreachable. When wiring a new tab or route, grep for all files that validate/allowlist the identifier (`VALID_TABS`, `NAV`, `activeKeyFor`, etc.), not just the one that renders it.

- **`@devdigest/ui`'s `Drawer` (`vendor/ui/kit/Drawer.tsx`) is the right component for any "open content in a right-side panel" need — already used by `RunTraceDrawer`.** When the Project Context list page needed a way to open a doc's content (previously only `Modal`-based previews existed, e.g. `PreviewModal` in the Agent Context tab), `Drawer` was the correct reuse target: it's a right-anchored, full-height slide-in panel (`position: fixed`, `justifyContent: "flex-end"`), not a centered dialog. Don't build a new right-sidebar pattern from scratch or reach for `Modal` when the ask is specifically a sidebar/drawer — check `vendor/ui/kit/` first.
  - **Correction (2026-07-10):** for the Project Context page specifically, `Drawer` was the *wrong* choice — the user wanted a true list-detail (master-detail) layout, first row auto-selected, content inline in a right column, not an overlay that pops over the list. Replaced with an inline detail panel mirroring `SkillsListView`'s existing master-detail shell (`skills/_components/SkillsListView/{styles,SkillsListView}.tsx`: `s.shell` = flex row, `s.list` = fixed-width left column, `s.detail` = flex-1 right column, no overlay). Lesson: "open content in a right-side panel" is ambiguous between an *overlay drawer* (temporary, dismissible, covers content) and a *master-detail column* (permanent, always-visible, part of the layout) — these look superficially similar in a one-line spec but are different components with different UX. When a page's job is fundamentally "browse a list and view one item's content," default to master-detail (check for an existing one first, e.g. `SkillsListView`) rather than `Drawer`, even though `Drawer` is the right choice for transient/secondary content elsewhere.
- **Making a whole row clickable while it also contains its own interactive control (a dropdown trigger) requires `stopPropagation` on the inner control's wrapper, not just relying on event target checks.** `DocRow`'s row got `onClick={() => onOpen(doc.path)}`, but it also renders a `Dropdown` trigger button for "Used by N agents" — without wrapping that in a `<div onClick={(e) => e.stopPropagation()}>`, clicking the dropdown would also fire the row's open handler (both fire on the same click, since dropdown trigger is a descendant). Verified live: clicking "Used by 1 agent" only opens its dropdown, not the drawer.
- **`@devdigest/ui`'s `Markdown` (`vendor/ui/primitives/Markdown.tsx`) is the existing, already-safe reusable markdown renderer — check for it before treating "render markdown" as a from-scratch task.** It wraps `react-markdown` + `remark-gfm` with theme-matched overrides for `p`/`strong`/`code`/`a`, and is exported from the package root (`export * from "./primitives"`). `DocDetail.tsx` was rendering `preview.data?.content` as a raw string (`<div>{content}</div>`) — swapping in `<Markdown>{content}</Markdown>` was a two-line change, not a new-dependency task (react-markdown/remark-gfm were already installed). It renders to React elements, not `dangerouslySetInnerHTML`, so it stays safe for untrusted repo content.

Confirmed as intentional by finding `PrBriefCard.tsx` already doing the same thing (`useTranslations("brief")` alongside a second call). - **Stack of identical toasts `Rate limit exceeded, retry in N seconds` on Export to CI → Preview (stuck on "Generating…")** → looks like GitHub API spam, but `CiService.preview` (`server/src/modules/ci/service.ts`) is explicitly non-mutating and makes **no** GitHub calls. Fix the client loop; do not chase `GITHUB_TOKEN` / octokit quotas for this symptom.
- **`getByRole("button", { name: "Close" })` throws `Found multiple elements` in any test that renders the shared kit `Modal`** → the `Modal` header always renders an `IconBtn icon="X" label="Close"` (aria-label `"Close"`, `vendor/ui/kit/Modal.tsx:58`), so a footer `<Button>Close</Button>` collides with it on accessible name (two buttons named "Close"). The header X button has no text child, so target the footer button with `getByText("Close")` (or scope with `within(footer)`), not `getByRole`. Same trap for any modal whose footer action word doubles as an icon-button aria-label. Hit while testing `CompareRunsModal`'s Close-only footer (AC-29).
- **Toggling an inline `style` object between a base style using shorthand `border` and a variant spreading a longhand `borderColor` leaves stale borders on deselected elements.** `DocRow.tsx` originally had `s.row = { border: "1px solid var(--border)", ... }` and `s.rowActive = { borderColor: "var(--border-strong)", ... }`, applied as `active ? {...s.row, ...s.rowActive} : s.row`. React's inline-style diffing sets/clears style keys by literal property name across renders — when an element goes from active (has key `borderColor`) back to inactive (only has key `border`), React clears `borderColor` *and* separately re-applies the `border` shorthand, but the browser's own shorthand-vs-longhand precedence left several rows with no visible border color at all (computed `border-color` fell back to an unrelated near-white default) after being deselected once. Confirmed via `getComputedStyle(el).borderColor` on rows that had previously been the selected row. Fix: use the same longhand form (`borderWidth`/`borderStyle`/`borderColor`) in both the base and active style objects so only `borderColor` ever changes between them — see `ProjectContextView/styles.ts` `s.row`/`s.rowActive`. General rule: any two inline-style objects meant to be merged/toggled on the same element must use identical property forms (all-shorthand or all-longhand) for any property that appears in both, never mix them.
- **Smart Diff showed extra/duplicate severity badges on unrelated deleted lines** — root cause: `SmartDiffViewer.tsx`'s `DiffLine` computed `lineNo = ln.newNo ?? ln.oldNo` and used that single value both to cross-reference findings (`findingForLine`) and to render the badge condition. `parsePatch` (`diff-viewer/helpers.ts`) only sets `oldNo` on `del` lines, never `newNo` — so a deleted line's *old-file* line number silently stood in for a *new-file* line number wherever `newNo` was absent. `finding.start_line`/`end_line` are always new-file coordinates, so any deleted line whose `oldNo` numerically coincided with a finding's new-file range got a false-positive stripe + badge. Confirmed concretely on PR #23's `NewTimeAttackButton.tsx` (single hunk `@@ -1,48 +1,100 @@`): finding at new-line 44 also lit up the deleted line at old-line 44. Fix: only pass `ln.newNo` (never fall back to `oldNo`) into `findingForLine` and the badge's `lineNo === finding.start_line` check; keep a separate `displayNo = ln.newNo ?? ln.oldNo` purely for the visible line-number column. Old-file and new-file line numbers are different coordinate spaces — never conflate them via `??` for anything that gets compared against review data (which is always new-file-anchored).
- **`textDecoration: underline` has no effect on SVG icons inside `display: inline-flex`** — root cause: `text-decoration` only draws under text nodes, not SVG children. Fix: use `borderBottom: "1px solid/dashed/dotted <color>"` + `paddingBottom: 2` on the `inline-flex` span instead — border applies to the element box and covers both icon and number. See `SeverityChips.tsx` interactive chip style.
- **`TS2322: Type 'number | null | undefined' is not assignable to type 'number | null'` in `RunHistory.test.tsx`** — root cause: `RunSummary` gained a new `nullable()` field (not `nullish()`), so the `Partial<RunSummary>` factory in the test fixture no longer satisfies the full type. Fix: add the new field with an explicit `null` default inside the `run()` factory function (`RunHistory.test.tsx:30`).
- **`Module not found: Can't resolve './contracts/findings.js'` at `src/vendor/shared/index.ts:17`, taking down EVERY route while `pnpm typecheck` stays green** — root cause: the vendored `@devdigest/shared` barrel is **type-only in practice**. All ~95 client imports from it are `import type`, so SWC erases them and webpack never has to resolve the barrel. The first *value* import — SPEC-03's `import { ExpectedFinding } from '@devdigest/shared'` in `EvalCaseModal/helpers.ts:2`, pulling in a Zod **schema** (a runtime object), not just a type — forces real resolution, and the barrel's ESM-style `.js` extensions (`export * from './contracts/findings.js'`) point at files that are actually `.ts`. `tsc` follows `.js` → `.ts` under its moduleResolution; Next's webpack does not. Fix: `config.resolve.extensionAlias = { '.js': ['.ts', '.tsx', '.js'] }` in `next.config.mjs`'s `webpack` hook — after which any shared Zod schema is importable as a value on the client. **The trap is the failure mode, not the fix**: typecheck, and every test that mocks the contracts, pass cleanly while the app is 100% dead on all routes — only loading a page in a browser reveals it. Whenever you add the first value import from a vendored barrel, load the app, don't just typecheck it.
- **A `z.string().min(1)` (or any `.min`/`.max`/`.regex`) constraint does NOT surface in the inferred TS type, so a constant annotated with the contract type can typecheck and still fail that contract's own runtime validation.** SPEC-03's `FINDING_SKELETON: ExpectedFinding` (`EvalCaseModal/constants.ts:8-14`) sets `title: ''`, but `ExpectedFinding.title` is `z.string().min(1)` (`vendor/shared/contracts/eval-ci.ts:42`) — `z.infer` widens both to plain `string`, so the annotation is satisfied. Live symptom: clicking the modal's `Finding skeleton` button inserts the skeleton and the field *immediately* goes red with "invalid JSON" and disables Save — on JSON the app itself just wrote, which is syntactically valid and failing only schema validation. Two lessons: (1) never assume a `T`-annotated literal satisfies `T`'s Zod schema — if a constant is meant to be a valid instance, `Schema.parse(CONST)` it in a test; (2) don't label a Zod `safeParse` failure "invalid JSON" — distinguish a `JSON.parse` throw (syntax) from a schema rejection (shape), or you send the user hunting for a missing comma that isn't there. Same family as the `api.get<T>` entry above: the generic/annotation is documentation, not a boundary check.
- **Giving a flex-column child both `flex: 1` and a fixed sibling `minHeight` on the shared ancestor makes it stretch to fill dead space instead of shrink-wrapping its own content.** `EvalCaseModal`'s two-column body (`styles.ts`'s `s.body`) had `minHeight: 480` plus `flex: 1` on `colPad`/`tabPanel` (left column) and on the right column's content wrapper. `Modal.tsx` itself has no fixed height (sizes to content up to `maxHeight: 92%`), so nothing actually required that 480px floor — but once it existed, every `flex: 1` descendant dutifully expanded to consume it, producing a huge blank gap between the "Input" label and the Diff/Files/PR meta tabs (and a similarly oversized empty JSON box) whenever the case was new/empty. A design mockup populated with real diff/JSON content masked the bug because its content happened to fill that same forced height. Fix: delete `minHeight: 480` from `s.body` and drop `flex: 1` from `colPad`, `tabPanel`, and the right column's padding wrapper (`EvalCaseModal.tsx`) — each section now sizes to its content (driven by `Textarea`'s `rows` prop), and `overflow: auto` alone still handles the case where content genuinely overflows. General rule: a `flex: 1` child only grows to fill genuinely *available* space — if a `minHeight`/fixed height anywhere upstream manufactures space that the actual content doesn't need, `flex: 1` will blindly stretch into it; check for that combination whenever a "why is there so much whitespace" bug shows up in a flex layout, and always test the *empty* state of a form/modal, not just a populated mockup.

- **`api.post<T>(path, body)` sends no `content-type` header at all — not even an empty JSON object — whenever `body` is falsy (`undefined`, `0`, `""`), which 422s any Fastify route whose Zod body schema is `z.object({...optional fields})`.** `apiFetch`/`api.post` (`src/lib/api.ts:32,72`) does `body ? JSON.stringify(body) : undefined`; with no body and no `content-type`, Fastify parses the request body as `null`, and a Zod schema like `RunCasesBody = z.object({ case_ids: z.array(...).optional() })` rejects `null` with `"Expected object, received null"` — even though every field inside it is optional. `EvalsTab.tsx`'s "Run all evals" called `runEvals.mutate(undefined)` to mean "no filter, run everything," which is exactly this trap; the per-case "Run" button worked because `{ case_ids: [id] }` is truthy. Fix: call `mutate({})` instead of `mutate(undefined)` — an empty object is truthy, so `apiFetch` sets `content-type: application/json` and sends `"{}"`, which satisfies the all-optional Zod object. General rule: when a POST/PUT body is a Zod object schema with all-optional fields (not `.optional()` on the whole schema), the client must send a real `{}`, never rely on `api.post`'s falsy-body branch to mean "empty object" — it means "no body sent."
- **Eval-dashboard MetricBars showed empty grey tracks next to correct `NN%` labels** — root cause: `fillStyle` (`MetricBar/styles.ts`) set `width: "${pct}%"` on a bare `<span>`, which is `display: inline`; percentage width is ignored on inline boxes, so the coloured fill collapsed to nothing while the track (explicit px width/height) and the separate label still rendered. The correct percentages are a red herring — they prove the 0–1 → `Math.round(value * 100)` path is fine, and will send you hunting missing CSS vars / wrong scale / null values when the bug is only layout. Fix: `display: "block"` on the fill (or use a `<div>` / `inline-block`). General rule: any inline-style "progress fill" as a child of a fixed-size track must be block-level, or width is a no-op.
