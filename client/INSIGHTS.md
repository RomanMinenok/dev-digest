## What Works
<!-- Approaches/solutions that worked. e.g. "X via Y in src/foo.ts:42 because ‚Ä¶" -->
- **Verifying a new page when you can't click through it: `pnpm build` + a static `t()`-key audit.** `pnpm typecheck` does NOT catch RSC/client-boundary errors or missing-i18n-key crashes; `next build` catches the former (compiles + generates every route) but still NOT the latter ‚Äî a missing `next-intl` `t("a.b")` key throws only at *render*, so it slips past both typecheck and build. The cheap guard is a node script that flattens `messages/en/<ns>.json` into dotted paths and checks every `t("‚Ä¶")` literal in the route's files resolves (treat `t(\`prefix.${x}\`)` template keys by their static prefix ‚Äî confirm a child exists). Used to validate the whole `/skills` tree after an agent built it without runtime checks.

## What Doesn't Work
<!-- Dead ends & antipatterns. The most valuable & most-skipped section. -->
- **A later, unrelated feature reintroduced an already-fixed scroll race by adding a THIRD actor that scrolls the same container.** `434c239` fixed Smart Diff's badge‚Üífinding-card centering by making `FindingsPanel` the sole scroll owner (deferred rAF + `ReviewRunAccordion` skipping its own `scrollIntoView` when `targetFindingId` is set). Commit `349aadd` later added a tab-scroll-persistence effect in `page.tsx` (`React.useEffect(() => { main.scrollTop = tabScrollTop.current[tab] ?? 0; ... }, [tab])`) that runs on every tab switch, including a badge-click switch to the findings tab ‚Äî it doesn't know about `findingId` and jumps `main.scrollTop` to a stale saved offset from the *previous* findings-tab visit before `FindingsPanel`'s smooth-centering scroll takes over. Confirmed via a `requestAnimationFrame` scrollTop trace: the buggy version jumps instantly to the stale offset then crawls back over ~1.1s to the correct centered position ‚Äî same final resting `scrollTop` as the fixed version, so a single before/after position check misses it entirely (see Tool & Library Notes entry below). Fix: `page.tsx`'s restore effect skips the jump when `tab === "findings" && findingId` (a pending target) is true, leaving `FindingsPanel` as sole owner ‚Äî same principle as the original fix, just applied to a new actor. **General rule: when adding any effect that sets `scrollTop`/`scrollIntoView` on a shared scroll container, grep for existing scroll owners on that same container before assuming there's only one.**
- **`AppShell` calls `useRouter()` internally (via `useGlobalShortcuts`) ‚Äî rendering it in RTL tests crashes with "invariant expected app router to be mounted".** The fix: mock the shell away entirely in the test file: `vi.mock("../../../../components/app-shell", () => ({ AppShell: ({ children }) => <>{children}</> }))`. Use this pattern for any page-level component test that wraps with `AppShell`; don't try to satisfy the router requirement (it pulls in too many Next.js internals). See `app/conventions/_components/ConventionsView/ConventionsView.test.tsx`.
- **Early-return branches that replace the full page layout for edge states break the visible chrome.** When a page has fixed structural elements (header, toolbar), handling loading/empty/error via separate `return (<AppShell>‚Ä¶stripped layout‚Ä¶</AppShell>)` branches causes those elements to disappear ‚Äî visually jarring and inconsistent with the design. Pattern: keep one render path with the full structure always mounted; vary only the content *inside* the card/list area for each state. Demonstrated in `app/conventions/_components/ConventionsView/ConventionsView.tsx` ‚Äî the multi-branch version broke the UI; collapsed to a single return fixed it.
- **`apiFetch` (`src/lib/api.ts`) force-sets `content-type: application/json` on any non-null body, which breaks `FormData` uploads** ‚Äî the browser must set `multipart/form-data; boundary=‚Ä¶` itself, and an explicit JSON content-type overrides it, so the server can't parse the parts. Fix: skip the JSON content-type when `init.body instanceof FormData` (do NOT set content-type at all for FormData). The skill-import upload (`api.upload` ‚Üí `POST /skills/import/preview`) depends on this.
- The client has its own vendored copy of shared contracts at `client/src/vendor/shared/contracts/` that is **not auto-synced** with `server/src/vendor/shared/contracts/`. Adding a field to the server copy alone passes server typecheck but breaks client typecheck. Always mirror changes to both copies (`trace.ts` and `platform.ts` were both affected in the cost feature).
- **`AgentCard`'s `skillCount` prop existed in the component signature (`AgentCard.tsx:22`) but was never passed at either call site** (`AgentsListView.tsx`, `agents/[id]/page.tsx`) ‚Äî the skill-count badge silently never rendered in the app despite a passing test that explicitly set the prop. A prop being tested in isolation doesn't guarantee it's wired at the real call sites; grep all render call sites, not just the component file, when a "why doesn't X show up" bug turns out to be a missing field.

- **Two nested components each firing their own `scrollIntoView()` on the same target change race and land on a random final position.** `ReviewRunAccordion` scrolled its whole container (`block: "start"`) the instant it flipped `open` (before the newly-mounted child had laid out), while its child `FindingsPanel` separately scrolled the target finding card (`block: "start"`) right after mounting ‚Äî both effects fired off the same `targetFindingId`/`targetRunId` change, and the accordion's expand animation shifted layout mid-scroll. Symptom: clicking a Smart Diff badge sometimes landed the card off-screen. Fix: only one component should own the scroll for a given navigation ‚Äî `ReviewRunAccordion` now skips its own `scrollIntoView` whenever a `targetFindingId` is present (`ReviewRunAccordion.tsx`), leaving `FindingsPanel` as the sole scroller, deferred one `requestAnimationFrame` so it scrolls after the accordion's expand has committed, and using `block: "center"` (not `"start"`) so short elements don't hug the viewport edge. When two ancestors both want to scroll to "roughly the same place" on the same state change, pick exactly one owner.

## Codebase Patterns
<!-- Conventions & architectural decisions, with the "why". -->
- **A written Development Plan's specific icon names and file paths are not guaranteed to exist ‚Äî verify against the actual repo before implementing, don't just trust the prose.** The Blast Radius plan (`docs/plan/blast_radius_plan.md`) specified icon `"GitFork"` for the new tab and file path `client/src/messages/en/prReview.json`; neither was correct ‚Äî `client/src/vendor/ui/icons.tsx` has no `GitFork` in its lucide-react import list (had to substitute `"Workflow"`, an icon that IS registered), and the real i18n file lives at `client/messages/en/prReview.json` (no `src/`). Before wiring a plan-specified icon name into a `Tabs`/`Icon.X` usage, grep `client/src/vendor/ui/icons.tsx`'s export list first; before writing to a plan-specified file path, `find`/`ls` it first.
- **Two distinct POST endpoints for repo "refresh" do completely different things ‚Äî don't conflate them.** `POST /repos/:id/refresh` enqueues a heavy **repo clone job** (used by code indexing / repo-intel). `POST /repos/:id/poll` is a lightweight **PR-list sync from GitHub** (updates title, headSha, additions, deletions, filesCount, status). The UI Refresh button must call `/poll`; `useRefreshRepo` in `core.ts:82` was previously wired to `/refresh`, which is why pressing Refresh never updated the PR list. See `server/src/modules/polling/routes.ts` and `server/src/modules/repos/service.ts`.
- Components shared across a **route subtree** (parent + its `[param]` children) belong in the parent's `_components/` folder, not in `src/components/`. Example: `SeverityChips` is used by both `pulls/_components/PRRow` and `pulls/[number]/_components/RunHistory`, so it lives at `pulls/_components/SeverityChips/`. Only promote to `src/components/` when used across unrelated routes.
- **Lazy hover-fetch pattern** (React Query): use `enabled: hovered && !!id && hasData` to defer the network call until the user actually hovers over the trigger. Pair with `staleTime: 30_000` to avoid refetching on repeated hovers within 30 s. Implemented in `pulls/_components/PRRow/PRRow.tsx` for the findings popover ‚Äî avoids N requests on initial list render.
- **Mismatched hover zones between a block wrapper and an inline-flex child cause popover to never show** ‚Äî `PRRow` tracked `chipsHovered` on a full-width block `<div>` around `SeverityChips` to gate the lazy fetch. `SeverityChips` tracked its own `hovered` on the smaller `inline-flex` chips area to show the popover. The fetch fired from the large zone; the popover showed from the small zone ‚Äî they rarely overlapped in practice. Fix: fetch eagerly (gated by `hasFindings`, not by hover) so findings are already loaded when the user hovers the chips. See `PRRow.tsx` ‚Äî `enabled: !!pr.id && hasFindings` with no hover gate.
- **Moving hover state inside a component breaks callers that depended on external hover** ‚Äî `RunSeverityChips` used `enabled: hovered` for lazy React Query fetch, where `hovered` came from its own `useState`. When the hover zone (+ popover) moved inside `SeverityChips`, `RunSeverityChips` lost the hover signal and the fetch never fired. Fix: move the data fetch one level up to `RunHistory` (one `useQuery` per PR page, keyed by `prId`) and pass findings down as props. React Query deduplication still guarantees a single network request even if multiple child components share the same `queryKey`. See `RunHistory.tsx:findingsByRunId`.
- **Absolutely-positioned popovers inside table rows are clipped by `overflow: hidden`** ‚Äî `FindingsPopover` used `position: absolute` relative to `SeverityChips`'s wrapper, which is inside a PR list row. The row (or a grid ancestor) has `overflow: hidden`, so the popover was cut off. Fix: use `createPortal(card, document.body)` + `position: fixed` with coordinates from `ref.current.getBoundingClientRect()` captured on mouseenter. The portal escapes all clipping ancestors. See `FindingsPopover.tsx` + `SeverityChips.tsx:handleMouseEnter`.
- **`pointerEvents: none` on hover popovers** ‚Äî when a popover is absolutely positioned over its trigger and the popover itself doesn't need to be interactive, set `pointerEvents: none`. Without it, moving the mouse from the trigger div into the popover card fires `mouseLeave` on the trigger and immediately hides the popover, causing flicker. See `FindingsPopover.tsx:44`. If the popover needs to be interactive (links, buttons), extend the hover zone to cover both trigger + popover instead.
- **Filter badge counts must come from the raw (unfiltered) array, not the filtered result** ‚Äî in `FindingsPanel`, `severityCounts` is computed from `findings` (the full prop), not from `shown` (the filtered output). If it used `shown`, clicking a chip would mutate the counts on all other chips to reflect only the filtered subset, making them jump around. Computing from the source array keeps chip labels stable regardless of the active filter. See `FindingsPanel.tsx:32-39`.
- **Dynamic `Icon` component lookup requires two casts** ‚Äî `Icon[name as keyof typeof Icon] as React.ComponentType<{ size?: number }>`. The first cast (`keyof typeof Icon`) satisfies the TS index signature; the second (`as React.ComponentType<...>`) is needed because the namespace type is a union of all icon shapes and TS cannot infer JSX callability from it. Without both casts you get either an index error or "JSX element type does not have any construct or call signatures". See `FindingsPanel.tsx:71`.
- **`useActiveRepo()` from `lib/repo-context.tsx` is the correct hook for any page that needs the active repo** ‚Äî returns `{ repoId, activeRepo, reposLoaded }`, already synced with the top-left repo selector. Do not call `useRepos()[0]` directly (misses URL-path priority and localStorage persistence). Edge-state pattern: while `!reposLoaded` show a skeleton; when loaded and `!repoId` show an empty state ("Connect a repo first"). See `app/conventions/_components/ConventionsView/ConventionsView.tsx` for the implemented pattern.
- **Course lesson scaffolding: check `helpers.ts` and `messages/en/` before wiring a new route** ‚Äî When adding a route that corresponds to a future lesson (conventions, eval-dashboard, etc.), `client/src/components/app-shell/helpers.ts:activeKeyFor` likely already returns the correct sidebar key for that path, and a skeleton i18n file may already exist under `client/messages/en/`. Found both pre-wired for `/conventions` before adding any code. Creating duplicates or overwriting them causes subtle drift, so always read those two locations first.
- **`_components/` promotion to `src/components/` is skipped when the refactor would exceed the task scope** ‚Äî the rule "promote to `src/components/` when used across unrelated routes" yields to CLAUDE.md's "no refactor beyond what the task requires." Result: `CreateSkillModal` (under `skills/_components/`) is cross-imported by `ConventionsView` without being moved. The cross-import is intentional, not a bug. If a future task widens the sharing further, that is the moment to promote. See `app/conventions/_components/ConventionsView/ConventionsView.tsx:11`.

- **Convention title edits are client-side-only state, mirroring accept/reject ‚Äî there is no `PATCH /repos/:repoId/conventions/:id` endpoint.** `ConventionsView.tsx` keeps an `edits: Record<string, string>` map (id ‚Üí overridden title) alongside `statuses`; `toConvention()` applies the override for display, and `acceptedCandidates` applies it again before `buildSkillBody()` so the edited text reaches the generated skill. If a future lesson needs edits to survive a rescan/refresh, add the PATCH route in `server/src/modules/conventions/routes.ts` first ‚Äî right now a `rescan` overwrites the query cache and silently drops any in-progress edits.

- **The PR detail page's scroll container is `<main>` (`overflow-y: auto`), not `window`** ‚Äî `window.scrollY` stays `0` even when the page visibly scrolls, because `<main>` (inside `AppShell`) is the actual scrollable element (`clientHeight` ‚â™ `scrollHeight`; `<body>`/`<html>` are non-scrolling). Any manual verification of scroll behavior (browser devtools, tests) must check `document.querySelector('main').scrollTop`, not `window.scrollY`. `Element.scrollIntoView()` itself works fine either way ‚Äî it walks up to the nearest scrollable ancestor automatically ‚Äî but *checking* whether it worked requires reading the right property. Found while verifying Smart Diff's finding-click-to-scroll feature.
- **`router.replace(url)` in the Next.js App Router scrolls to top by default ‚Äî it fights a manual `scrollIntoView()` triggered by the same navigation.** Built a "click a badge on one tab ‚Üí switch tab + scroll to a specific card" flow (Smart Diff ‚Üí Findings tab, `_components/DiffTab`/`FindingsPanel`); the tab-switch `router.replace()` and the target card's `element.scrollIntoView()` effect both fired on the same navigation, and Next's own scroll-restoration always won, landing on the first card instead of the target. Fix: pass `router.replace(url, { scroll: false })` for that specific navigation so the app's own scroll effect is the only one acting. Don't disable `scroll` globally ‚Äî plain tab switches (Overview ‚Üí Files changed) still want the default top-scroll.
- **Running `pnpm build` while `pnpm dev` is live on the same package corrupts the shared `.next/` directory** ‚Äî both commands write into `.next/` (dev writes incremental chunks, build does a full optimized rebuild), and running them concurrently leaves the dev server serving stale/missing vendor-chunk references (`Cannot find module './vendor-chunks/...'` runtime error on every route). Symptom is a full-page Next.js Runtime Error dialog, not a compile error. Fix: kill the dev process, `rm -rf .next`, restart `pnpm dev`. Avoid running `pnpm build` as a "just checking" step while a dev server for the same package is running ‚Äî use it only when the dev server is stopped, or accept the restart cost.

- **`diff-viewer`'s `parsePatch`/styles (`s.hunk`, `s.lineNo`, `s.lineText`, `lineRowFor`, `lineSignFor`, `chevronFor`, `AUTO_EXPAND_MAX_LINES`) are all exported and safely reusable outside the diff-viewer folder ‚Äî `CodeLine` itself is not** (it's tightly coupled to `DiffCommentApi`/comment threads via `commentTargetFor`/`InlineComposer`). When a new feature needs to render raw diff lines without commenting (e.g. Smart Diff's per-role file rows), reuse the exported helpers/styles directly and build a small local line renderer ‚Äî don't try to reuse `CodeLine` itself or pass it a fake/empty `commenting` prop. See `SmartDiffViewer/SmartDiffViewer.tsx`.
- **A route contract that omits a field (here `SmartDiffFile` has no `patch`) doesn't mean the UI can't show that content ‚Äî check if the data is already loaded elsewhere on the page first.** Built Smart Diff's file rows initially as a static summary (path + stat + badges) because the `/smart-diff` route's `SmartDiffFile` has no `patch`. That was a mistake: `usePullDetail` already fetches the full `PrFile[]` (with `patch`) for the same PR, and it's already passed into `DiffTab`. Fix: thread `files: PrFile[]` into `SmartDiffViewer` and cross-reference by `path` ‚Äî no backend/contract change needed. Before declaring a UI capability "not possible" because one API response lacks a field, check sibling data already on the page.
- **A written implementation plan is not "done" until checked against its own explicit spec language, not just "does it toggle/render."** The plan said file rows "mirror `FileCard`'s collapse pattern" (i.e. expand to show the diff), but the first pass only built a collapsed summary row with no way to see the diff at all ‚Äî a real functional gap, not a polish issue. It passed typecheck, build, unit tests, and even a live browser click-through (toggle + badge navigation) because none of those checks compare rendered output against the actual target (a mockup screenshot, in this case). When a plan references a specific existing component's *behavior* (not just its visual style), verify the new component actually replicates that behavior, not just its collapsed-state appearance.

- **A single "find the finding for this line" helper can't correctly serve both range-highlighting and single-badge-placement ‚Äî they need different match semantics from the same call site.** Smart Diff's per-line severity stripe needed to span a finding's whole `start_line..end_line` range, but the clickable badge must render exactly once (at the top of the range), not once per matched line. Initially matched `line === start_line` only (so both stripe and badge were correct only on line 1 of the range, and the stripe never covered the rest). Fix: `findingForLine` (`SmartDiffViewer/helpers.ts`) matches the **inclusive range** (`start_line <= line <= end_line`) for stripe color; the caller (`SmartDiffViewer.tsx`'s `DiffLine`) additionally checks `lineNo === finding.start_line` before rendering the badge. One helper, one range-based match, two different uses of the result ‚Äî don't try to encode "and also only on the first line" into the matching function itself.

- **A server-side session-window filter (Decision 2's 60s rule) doesn't automatically apply to every client-side consumer of the same findings ‚Äî each place that fans a review list out into a flat findings list needs its own filter, because `FindingRecord` has no `created_at` of its own.** `assembleSmartDiff` correctly filters `reviews` to the 60s session window server-side before building `finding_lines` (used for the collapsed-row indicator dot in `SmartDiffViewer`). But `page.tsx`'s `allFindings = runs.flatMap(r => r.findings)` (used for the Findings tab / lethal-trifecta banner, intentionally *not* session-windowed ‚Äî it should show every finding ever recorded) was also being passed to `SmartDiffViewer` for the per-line stripe/badge rendering, silently including stale out-of-session findings there too ‚Äî a real behavioral mismatch caught only by re-reading the plan's Decision 2 against the actual data flow, not by any test or visual check. Fix: `sessionWindowFindings(reviews)` (`SmartDiffViewer/helpers.ts`) mirrors the server logic (filter reviews by 60s window, then flatten) and feeds a *separate* `smartDiffFindings` value distinct from `allFindings`. When a decision says "X window applies to feature Y," check every place Y's data is derived, not just the one where the rule was first implemented.

- **A tab's scroll position must be persisted in the parent, not the tab's own component, because the tab unmounts on switch.** `page.tsx` renders `tab === "diff" && <DiffTab/>` (conditional mount, not CSS-hidden), so any `useState` scroll tracking inside `DiffTab`/`SmartDiffViewer` is destroyed the instant the user switches tabs. Fix: keep a `React.useRef(0)` in `page.tsx` itself (survives across tab switches since the page component never unmounts), attach a `scroll` listener to `document.querySelector('main')` (the real scroll container ‚Äî see the `<main>`-not-`window` entry below) only while `tab === "diff"`, and restore `main.scrollTop` from the ref in an effect keyed on `tab` whenever it re-becomes `"diff"`. Built for the Smart-Diff-badge round trip (diff ‚Üí findings ‚Üí back to diff should land exactly where the user left off), see `page.tsx`'s `diffScrollTop` ref.

- **A one-shot "target and animate to it" URL param (`findingId`) must be cleared by the consumer once acted on, or every later remount replays the animation.** `page.tsx`'s `tabScrollTop` ref (extended from the diff-only `diffScrollTop` in the entry above to a `Record<tab, number>` covering both `"diff"` and `"findings"`) now restores raw scroll position on any tab revisit ‚Äî but `findingId` stayed in the URL indefinitely (nothing ever cleared it), so every remount of `FindingsTab`/`ReviewRunAccordion`/`FindingsPanel` still saw a non-null `targetFindingId` and re-ran the scroll-to-card `useEffect`, re-triggering the smooth-scroll+center animation on top of the just-restored position on every single tab revisit, not just the first. Fix: `FindingsPanel` calls a new `onScrolledToTarget` callback (bubbled through `ReviewRunAccordion` ‚Üí `FindingsTab` ‚Üí `page.tsx`) right after firing its one-shot `scrollIntoView`, and `page.tsx` uses it to drop `findingId` from the URL (`setParams({ findingId: null }, { scroll: false })`). Only the panel that actually owns the finding (`shown.findIndex(...) >= 0`) calls it, so panels for other review runs don't prematurely clear the target before the right one has had a chance to scroll. General rule: any prop meant to trigger a one-time effect on arrival (not a fact to remember forever) needs an explicit "consumed" signal back to whoever owns that prop's source of truth ‚Äî otherwise every future remount re-fires it.

## Codebase Patterns (addendum)
- **Sidebar wiring is split across TWO separate files, and only one of them makes a route actually clickable.** `components/app-shell/helpers.ts`'s `activeKeyFor` only decides which sidebar key is *highlighted* for a given path ‚Äî it does NOT add an entry to the sidebar itself. The actual list of clickable sidebar items lives in `client/src/vendor/ui/nav.ts` (a `NAV` array). Adding a new route and only updating `activeKeyFor` (as the project-context implementation plan's T14 task literally scoped it) leaves the new page unreachable from the UI ‚Äî the route works if you type the URL directly, but there's no link to it. When wiring a brand-new top-level route into the sidebar, update `nav.ts` as well as `activeKeyFor`, even if a task/plan only names the latter.

- **Adding a new tab to `AgentEditor` requires updating a SECOND, separate file: `app/agents/[id]/page.tsx`'s `VALID_TABS` allowlist.** `AgentEditor.tsx`'s own tab switch/`TABS` array is not the only gate ‚Äî `page.tsx:15` (`const VALID_TABS = ["config", "skills"]`) validates the `?tab=` query param before it ever reaches `AgentEditor`, silently falling back to `"config"` for any tab name not in that array. Adding T15's new `"context"` tab to `AgentEditor.tsx`/`constants.ts` alone left the tab fully built but unreachable via URL/deep-link. Same *shape* of bug as the `nav.ts`/`activeKeyFor` sidebar split noted above ‚Äî two independent gate files for one feature, and a plan/task that only names one of them will compile clean and look "done" while being unreachable. When wiring a new tab or route, grep for all files that validate/allowlist the identifier (`VALID_TABS`, `NAV`, `activeKeyFor`, etc.), not just the one that renders it.

- **`@devdigest/ui`'s `Drawer` (`vendor/ui/kit/Drawer.tsx`) is the right component for any "open content in a right-side panel" need ‚Äî already used by `RunTraceDrawer`.** When the Project Context list page needed a way to open a doc's content (previously only `Modal`-based previews existed, e.g. `PreviewModal` in the Agent Context tab), `Drawer` was the correct reuse target: it's a right-anchored, full-height slide-in panel (`position: fixed`, `justifyContent: "flex-end"`), not a centered dialog. Don't build a new right-sidebar pattern from scratch or reach for `Modal` when the ask is specifically a sidebar/drawer ‚Äî check `vendor/ui/kit/` first.
  - **Correction (2026-07-10):** for the Project Context page specifically, `Drawer` was the *wrong* choice ‚Äî the user wanted a true list-detail (master-detail) layout, first row auto-selected, content inline in a right column, not an overlay that pops over the list. Replaced with an inline detail panel mirroring `SkillsListView`'s existing master-detail shell (`skills/_components/SkillsListView/{styles,SkillsListView}.tsx`: `s.shell` = flex row, `s.list` = fixed-width left column, `s.detail` = flex-1 right column, no overlay). Lesson: "open content in a right-side panel" is ambiguous between an *overlay drawer* (temporary, dismissible, covers content) and a *master-detail column* (permanent, always-visible, part of the layout) ‚Äî these look superficially similar in a one-line spec but are different components with different UX. When a page's job is fundamentally "browse a list and view one item's content," default to master-detail (check for an existing one first, e.g. `SkillsListView`) rather than `Drawer`, even though `Drawer` is the right choice for transient/secondary content elsewhere.
- **Making a whole row clickable while it also contains its own interactive control (a dropdown trigger) requires `stopPropagation` on the inner control's wrapper, not just relying on event target checks.** `DocRow`'s row got `onClick={() => onOpen(doc.path)}`, but it also renders a `Dropdown` trigger button for "Used by N agents" ‚Äî without wrapping that in a `<div onClick={(e) => e.stopPropagation()}>`, clicking the dropdown would also fire the row's open handler (both fire on the same click, since dropdown trigger is a descendant). Verified live: clicking "Used by 1 agent" only opens its dropdown, not the drawer.
- **`@devdigest/ui`'s `Markdown` (`vendor/ui/primitives/Markdown.tsx`) is the existing, already-safe reusable markdown renderer ‚Äî check for it before treating "render markdown" as a from-scratch task.** It wraps `react-markdown` + `remark-gfm` with theme-matched overrides for `p`/`strong`/`code`/`a`, and is exported from the package root (`export * from "./primitives"`). `DocDetail.tsx` was rendering `preview.data?.content` as a raw string (`<div>{content}</div>`) ‚Äî swapping in `<Markdown>{content}</Markdown>` was a two-line change, not a new-dependency task (react-markdown/remark-gfm were already installed). It renders to React elements, not `dangerouslySetInnerHTML`, so it stays safe for untrusted repo content.

## Tool & Library Notes
<!-- Dependency quirks, version gotchas, env/config oddities. -->
- `NEXT_PUBLIC_*` env vars are inlined into the bundle at **start time** (`next dev` / `next build`), not at request time. Changing a `NEXT_PUBLIC_*` value in `.env` requires a dev-server restart to take effect ‚Äî a page refresh is not enough.
- **`chrome-devtools-mcp`'s `take_snapshot` on a large expanded Smart Diff view (23 files, full patches) blows the tool's ~130k-char response limit and errors instead of returning.** For pages with large DOM trees, don't rely on `take_snapshot` for verification ‚Äî use `evaluate_script` to read/manipulate specific state instead (e.g. `document.querySelector('main').scrollTop`, or `document.querySelector('button[aria-label^="..."]').click()` found via a `TreeWalker` text search). Also: `navigate_page` does a hard browser navigation and resets all in-page React/ref state ‚Äî to test client-side tab navigation (e.g. verifying a `useRef` survives a tab switch), trigger the actual in-app click via `evaluate_script`, not `navigate_page` with a new `?tab=` URL.

## Recurring Errors & Fixes
<!-- Error signature ‚Üí root cause ‚Üí fix. -->
- **Toggling an inline `style` object between a base style using shorthand `border` and a variant spreading a longhand `borderColor` leaves stale borders on deselected elements.** `DocRow.tsx` originally had `s.row = { border: "1px solid var(--border)", ... }` and `s.rowActive = { borderColor: "var(--border-strong)", ... }`, applied as `active ? {...s.row, ...s.rowActive} : s.row`. React's inline-style diffing sets/clears style keys by literal property name across renders ‚Äî when an element goes from active (has key `borderColor`) back to inactive (only has key `border`), React clears `borderColor` *and* separately re-applies the `border` shorthand, but the browser's own shorthand-vs-longhand precedence left several rows with no visible border color at all (computed `border-color` fell back to an unrelated near-white default) after being deselected once. Confirmed via `getComputedStyle(el).borderColor` on rows that had previously been the selected row. Fix: use the same longhand form (`borderWidth`/`borderStyle`/`borderColor`) in both the base and active style objects so only `borderColor` ever changes between them ‚Äî see `ProjectContextView/styles.ts` `s.row`/`s.rowActive`. General rule: any two inline-style objects meant to be merged/toggled on the same element must use identical property forms (all-shorthand or all-longhand) for any property that appears in both, never mix them.
- **Smart Diff showed extra/duplicate severity badges on unrelated deleted lines** ‚Äî root cause: `SmartDiffViewer.tsx`'s `DiffLine` computed `lineNo = ln.newNo ?? ln.oldNo` and used that single value both to cross-reference findings (`findingForLine`) and to render the badge condition. `parsePatch` (`diff-viewer/helpers.ts`) only sets `oldNo` on `del` lines, never `newNo` ‚Äî so a deleted line's *old-file* line number silently stood in for a *new-file* line number wherever `newNo` was absent. `finding.start_line`/`end_line` are always new-file coordinates, so any deleted line whose `oldNo` numerically coincided with a finding's new-file range got a false-positive stripe + badge. Confirmed concretely on PR #23's `NewTimeAttackButton.tsx` (single hunk `@@ -1,48 +1,100 @@`): finding at new-line 44 also lit up the deleted line at old-line 44. Fix: only pass `ln.newNo` (never fall back to `oldNo`) into `findingForLine` and the badge's `lineNo === finding.start_line` check; keep a separate `displayNo = ln.newNo ?? ln.oldNo` purely for the visible line-number column. Old-file and new-file line numbers are different coordinate spaces ‚Äî never conflate them via `??` for anything that gets compared against review data (which is always new-file-anchored).
- **`textDecoration: underline` has no effect on SVG icons inside `display: inline-flex`** ‚Äî root cause: `text-decoration` only draws under text nodes, not SVG children. Fix: use `borderBottom: "1px solid/dashed/dotted <color>"` + `paddingBottom: 2` on the `inline-flex` span instead ‚Äî border applies to the element box and covers both icon and number. See `SeverityChips.tsx` interactive chip style.
- **`TS2322: Type 'number | null | undefined' is not assignable to type 'number | null'` in `RunHistory.test.tsx`** ‚Äî root cause: `RunSummary` gained a new `nullable()` field (not `nullish()`), so the `Partial<RunSummary>` factory in the test fixture no longer satisfies the full type. Fix: add the new field with an explicit `null` default inside the `run()` factory function (`RunHistory.test.tsx:30`).
